    🤖 AI Ownership Matrix - Distributed Responsibility

    AI capabilities are distributed across team members based on their technical strengths, ensuring multiple team members understand and own AI features.

    Iman Wael (Team Leader) - AI Orchestration & Strategic Services

    Mustafa (Frontend Lead) - AI UI & User Experience

    Primary AI Responsibilities:
    • Display AI-generated reports & summaries
    • Chat interface for AI support
    • Voice memo recording UI
    • Loading states & AI processing feedback
    Rationale: As frontend lead, creates user-friendly interfaces for AI features. Ensures AI outputs are presented clearly and chat is responsive.

    Yara (Full Stack Developer A) - Patient Data & OCR

    Primary AI Responsibilities:
    • Upload Intake Form Image & OCR integration

    Mohammed (Full Stack Developer B) - Voice & Conversational AI

    • Proper segregation of concerns (OCR, Voice, Chatbot, Decision Support)

    Focus: Authentication, database design, core APIs, UI framework

    Iman Wael (Team Leader): AI Services Setup & Patient Dashboard Skeleton

    🔗 Integration Plan & Dependency Analysis

    Critical Integration Flows

    🩺 Patient Intake & OCR Integration

    Patient uploads intake form image
    ↓
    Full Stack A: Image sent to Team Leader's OCR service
    ↓
    Team Leader: Vision API extracts structured data
    ↓
    Full Stack A: Validates & stores in Patient profile
    ↓
    ✓ Patient intake form complete

    🎤 Voice Memo & Transcription Pipeline

    Therapist records voice memo in session note
    ↓
    Frontend Lead: Records audio via browser API
    ↓
    Full Stack B: Uploads audio to storage
    ↓
    Full Stack B: Sends to Team Leader's Whisper API
    ↓
    Team Leader: Returns transcription
    ↓
    Full Stack B: Stores transcript in session note
    ↓
    ✓ Voice memo transcribed

    📋 Report Generation Pipeline (Core AI Feature)

    Therapist requests to generate referral report
    ↓
    Team Leader: Fetches patient history, assessments, sessions
    ↓
    Team Leader: Uses RAG to retrieve relevant session context
    ↓
    Team Leader: Generates report draft via GPT-4o
    ↓
    Frontend Lead: Displays report for review
    ↓
    Therapist: Edits, approves, or rejects report
    ↓
    Backend Lead: Exports to PDF on approval
    ↓
    ✓ Report generation complete

    💬 Patient Chat with AI & Crisis Detection

    Patient sends message in support chat
    ↓
    Frontend Lead: WebSocket sends message to Backend
    ↓
    Backend Lead: Stores message, retrieves conversation history
    ↓
    Full Stack B: Sends to GPT-4o for response generation
    ↓
    Team Leader: Analyzes message for crisis keywords (NLP)
    ↓
    IF crisis detected → Backend Lead: Triggers crisis alert
    ↓
    Backend Lead: Sends notification to therapist via Notification Service
    ↓
    Frontend Lead: Displays response & crisis alert (if needed)
    ↓
    ✓ Chat message processed with safety monitoring

    📊 Analytics & Dashboard Data Aggregation

    Therapist opens patient progress dashboard
    ↓
    Frontend Lead: Requests analytics from Team Leader's endpoint
    ↓
    Team Leader: Aggregates assessments, exercises, sessions
    ↓
    Full Stack A: Provides exercise completion statistics
    ↓
    Full Stack B: Provides session statistics
    ↓
    Frontend Lead: Renders charts & visualizations
    ↓
    ✓ Dashboard displays consolidated patient progress

    Business Domain Driven Architecture

    Key Message:

    Instead of organizing by technical layers (Frontend/Backend/Database), we've organized Jalsa by 8 business domains that map directly to the UML Use Case Diagram. This ensures:

    • Requirements traceability: Every task is traceable to a use case
    • Business value: Team understands what users need, not just technical tasks
    • Reduced rework: Clear ownership prevents duplicate/conflicting work
    • Easier testing: Each domain has clear acceptance criteria from use cases

    Fair Full-Stack Distribution

    Key Message:

    Every team member owns complete business features from database to UI, not just one technical layer. This ensures:

    • Cross-functional expertise: Everyone learns full-stack development
    • No bottlenecks: Each member can work independently
    • Knowledge distribution: Knowledge isn't siloed to one person
    • Career growth: Team members develop diverse skills

    AI Strategy

    Key Message:

    Rather than having one person "own AI," we've distributed AI features based on technical fit:

    • Iman (Team Leader): Strategic AI (Reports, crisis detection, RAG)
    • Salma (Backend Lead): AI infrastructure (RAG execution, API gateway, crisis alerts)
    • Mustafa (Frontend Lead): AI UI (report display, chat interface, voice recording UI)
    • Mohammed (Full Stack B): Conversational AI (Chatbot, voice/STT)
    • Yara (Full Stack A): Data extraction (OCR integration)
    • Result: Multiple team members understand AI, reducing knowledge risk

    Sprint Planning Philosophy

    Key Message:

    Sprints are organized by business value & dependencies, not technical layers:

    • Sprint 1: Build foundation (auth, patient base, sessions)
    • Sprint 2: Core features (intake forms, exercises, AI basics)
