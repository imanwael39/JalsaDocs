📝 Session Management – Phase 4

Purpose: Enable therapists to record session notes, upload voice memos, and generate AI‑powered summaries. This phase builds the backend and frontend for session‑related features, integrating with Whisper STT and GPT‑4o summarization.

📋 Tasks: 15 (9 backend + 6 frontend) ⏱️ Total Effort: ~68 hours 👤 Owners: M2 (Backend) + M3 (Frontend Lead) 🔗 Dependencies: Phase 3 (Patient Management) complete
BE-SES-001Session DTOs P0 Low 3h ▾

Task Information

  • Task ID: BE-SES-001
  • Task Name: Session DTOs
  • Owner: M2 (Backend)
  • Dependencies: Phase 1 – SessionNote, VoiceMemo, AISummary entities exist
  • Complexity: Low
  • Estimated Effort: 3 hours

Objective

Create DTOs for session operations: SessionNoteCreateDto, SessionNoteUpdateDto, SessionNoteViewDto, and SessionSearchDto.

Business Purpose

Define the structure of data exchanged between frontend and backend for session notes, ensuring proper validation and separation from domain entities.

Technical Purpose

Provide type‑safe contracts for API endpoints and enable mapping between entities and DTOs using AutoMapper.

Prerequisites

Backend

  • Data Annotations / FluentValidation: For validation.
  • DTO pattern: Understand role in Clean Architecture.

Inputs

  • SessionNote, VoiceMemo, AISummary entities from Phase 1.

Outputs

  • SessionNoteCreateDto – fields: PatientId, TherapistId (inferred from auth), Content, SessionDate, Status (Draft/Final).
  • SessionNoteUpdateDto – similar but includes Id.
  • SessionNoteViewDto – includes Id, PatientId, TherapistId, Content, SessionDate, Status, VoiceMemoUrl, Transcript, AISummary, CreatedAt, UpdatedAt.
  • SessionSearchDto – PatientId, DateFrom, DateTo, Status.

Detailed Backend Workflow

  1. Create DTO classes in Jalsa.Application/DTOs/Session/.
  2. Add validation attributes (e.g., [Required] for Content, [MaxLength(10000)]).
  3. For ViewDto, include nested objects (VoiceMemoDto, AISummaryDto) or flatten.

Deliverables

  • Four DTO classes with validation.

Validation Checklist

  • Required fields marked.
  • String lengths constrained.
  • Date validation (if needed).

Testing Strategy

Unit Tests: Validate DTOs with DataAnnotations or FluentValidation tests.

Definition of Done

  • DTOs created and committed.

Learning Resources

Must Know: DataAnnotations, DTO usage in Clean Architecture.

BE-SES-005Voice Memo – Upload P1 Medium 4h ▾

Task Information

  • Task ID: BE-SES-005
  • Task Name: Voice Memo – Upload
  • Owner: M2 (Backend)
  • Dependencies: BE-SES-003 (SessionService), Blob storage service
  • Complexity: Medium
  • Estimated Effort: 4 hours

Objective

Implement endpoint to upload an audio file (voice memo) for a session note, store it in blob storage, and create a VoiceMemo entity.

Business Purpose

Allow therapists to record audio during sessions and attach to session notes.

Technical Purpose

Handle file upload, validate format and size, upload to blob, save URL and duration to database.

Prerequisites

  • File upload: IFormFile.
  • Blob storage: Azure.Storage.Blobs or AWS S3.

Inputs

  • SessionNote ID, audio file (multipart/form-data).

Outputs

  • VoiceMemo entity created with AudioUrl, Duration (if available).

Detailed Backend Workflow

  1. Create endpoint POST /api/sessions/{sessionId}/voicememo.
  2. Accept IFormFile (name="audio").
  3. Validate file type (audio/mpeg, audio/wav, audio/x-m4a) and size (max 50MB).
  4. Generate unique filename (e.g., voicememo_{sessionId}_{timestamp}.mp3).
  5. Upload to blob storage, get URL.
  6. Create VoiceMemo entity: set SessionNoteId, AudioUrl, Duration (if can extract).
  7. Save via repository (or UnitOfWork).
  8. Return VoiceMemoViewDto.

Detailed Database Workflow

  • VoiceMemo table stores the URL and duration.

API Design

RouteMethodRequestResponse
/api/sessions/{sessionId}/voicememoPOSTmultipart/form-data (file)VoiceMemoViewDto

Data Flow

Client → Controller → Blob Storage → Database

Deliverables

  • Upload endpoint.
  • Blob storage integration.

Validation Checklist

  • File type validation.
  • File size limit.
  • URL saved correctly.

Testing Strategy

Integration Tests: Test file upload with mock blob.

Definition of Done

  • Upload works.
BE-SES-006Voice Memo – STT Integration P1 High 5h ▾

Task Information

  • Task ID: BE-SES-006
  • Task Name: Voice Memo – STT Integration
  • Owner: M5 (AI/DevOps)
  • Dependencies: BE-SES-005 (upload), BE-AI-002 (Whisper integration)
  • Complexity: High
  • Estimated Effort: 5 hours

Objective

After voice memo upload, trigger transcription using Whisper STT and store the transcript in the VoiceMemo entity.

Business Purpose

Transcribe therapist voice memos into text for later use in session notes and AI summarization.

Technical Purpose

Call the Whisper API with the audio URL or binary, parse the response, and update the VoiceMemo record.

Prerequisites

  • Whisper API (OpenAI or Azure).
  • Background job processing (Hangfire/Quartz) to avoid blocking.

Inputs

  • VoiceMemo ID (or SessionNoteId).
  • Audio URL or file.

Outputs

  • VoiceMemo.Transcript populated with transcribed text.

Detailed Backend Workflow

  1. After upload, enqueue a background job (Hangfire) to process transcription.
  2. Job fetches the audio URL, downloads the file (or sends URL to Whisper).
  3. Call Whisper API with the audio.
  4. Receive transcript string.
  5. Update VoiceMemo.Transcript and save.
  6. Optionally update SessionNote status or notify via WebSocket.

Detailed AI Workflow

Upload → Hangfire Job → Whisper API → Transcript → Database

API Design

No direct endpoint – triggered automatically after upload or via separate call.

Deliverables

  • Background job for transcription.
  • Whisper integration.

Validation Checklist

  • Transcription completes successfully.
  • Transcript stored in database.
  • Error handling for API failures.

Common Pitfalls

Pitfall: Blocking the request – always use background jobs for AI processing.

Definition of Done

  • Transcription works.
BE-SES-007AI Session Summary P1 High 6h ▾

Task Information

  • Task ID: BE-SES-007
  • Task Name: AI Session Summary
  • Owner: M2 (Backend) with AI support
  • Dependencies: BE-SES-006 (transcript), BE-AI-003 (summarization), Patient history
  • Complexity: High
  • Estimated Effort: 6 hours

Objective

Generate an AI summary for a session note using GPT‑4o, combining the session transcript and patient history.

Business Purpose

Provide therapists with a concise summary of the session to save time and improve clinical notes.

Technical Purpose

Call GPT‑4o with the transcript and relevant patient context (from RAG) to produce a structured summary.

Prerequisites

  • OpenAI API (GPT‑4o).
  • RAG retrieval to get patient history context.
  • Background jobs.

Inputs

  • SessionNote ID, transcript, patient ID.

Outputs

  • AISummary entity with SummaryText and GeneratedAt.

Detailed Backend Workflow

  1. Trigger summary generation (manually or after transcription).
  2. Retrieve session note, patient profile, previous session summaries (via RAG).
  3. Build a prompt: "Summarize this session: [transcript]. Patient context: [context]. Generate a concise clinical summary."
  4. Call GPT‑4o with the prompt.
  5. Parse response and save as AISummary.
  6. Link to SessionNote.

Detailed AI Workflow

SessionNote → RAG Retrieval → Prompt → GPT‑4o → Summary

API Design

Endpoint: POST /api/sessions/{sessionId}/generate-summary (or auto).

Deliverables

  • Summary generation logic.
  • Background job (if async).

Validation Checklist

  • Summary is generated.
  • Stored in AISummary.
  • Error handling.

Definition of Done

  • Summary generation works.
