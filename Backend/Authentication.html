🔐 Phase 2 – Authentication

Purpose: Implement JWT‑based authentication with role‑based authorization (Therapist / Patient). This phase secures all API endpoints, enables user registration, login, password reset, and profile management. It also sets up the frontend authentication flow.

📋 Tasks: 10 ⏱️ Total Effort: ~42 hours 👤 Owners: M1 (Backend Lead) + M3 (Frontend Lead) 🔗 Dependencies: Phase 1 (Database & Core Infra) must be complete
BE-AUTH-001 JWT Configuration P0 Medium 5h ▾

Task Information

  • Task ID: BE-AUTH-001
  • Task Name: JWT Configuration
  • Owner: M1 (Backend Lead)
  • Dependencies: Phase 1 – User and Role entities exist, DbContext and repositories ready
  • Complexity: Medium
  • Estimated Effort: 5 hours

Objective

Configure JWT authentication in the ASP.NET Core API. Set up token generation, validation parameters, and role‑based policies.

Business Purpose

Ensure only authenticated users (therapists and patients) can access protected endpoints. JWT provides a stateless, scalable authentication mechanism.

Technical Purpose

Integrate Microsoft.AspNetCore.Authentication.JwtBearer, define token parameters (issuer, audience, secret, expiry), and register the authentication middleware.

Prerequisites

Backend

  • JWT basics: Structure (header, payload, signature), claims, signing algorithms (HS256).
  • ASP.NET Core authentication: Middleware pipeline, AddAuthentication, AddJwtBearer.
  • Why needed: To correctly configure the JWT handler.
  • Usage: Every authenticated request will use this configuration.

Frontend

  • None – this is backend configuration.

Database

  • None – but the User table will be used to validate credentials.

Inputs

  • appsettings.json – JWT secret, issuer, audience, expiry minutes.
  • User and Role entities from Phase 1.

Outputs

  • JWT authentication middleware registered in Program.cs.
  • IJwtTokenGenerator interface and implementation.
  • Authorization policies for Therapist and Patient roles.

Detailed Backend Workflow

  1. Install NuGet package: Microsoft.AspNetCore.Authentication.JwtBearer.
  2. Add JWT settings to appsettings.json: 
    {
  "JwtSettings": {
    "Secret": "a-very-long-secret-key-at-least-32-characters",
    "Issuer": "JalsaAPI",
    "Audience": "JalsaClient",
    "ExpiryMinutes": 60
  }
}
    Why: Secret must be at least 32 characters for HS256. Keep it in environment variables in production.
  3. Create JwtSettings class in Jalsa.Infrastructure/Settings/ to bind configuration.
  4. Create IJwtTokenGenerator interface in Jalsa.Application/Interfaces/Auth/ with method string GenerateToken(User user, IList<string> roles).
  5. Implement JwtTokenGenerator in Jalsa.Infrastructure/Security/:
    • Read JwtSettings from DI.
    • Create SymmetricSecurityKey from Secret.
    • Create signing credentials (HS256).
    • Build claims: ClaimTypes.NameIdentifier (UserId), ClaimTypes.Email, ClaimTypes.Role.
    • Create JwtSecurityToken with issuer, audience, expiry, claims, signing credentials.
    • Write token using JwtSecurityTokenHandler.
    Why: Centralises token creation logic.
  6. Configure authentication in Program.cs: 
    builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer = true,
            ValidateAudience = true,
            ValidateLifetime = true,
            ValidateIssuerSigningKey = true,
            ValidIssuer = jwtSettings.Issuer,
            ValidAudience = jwtSettings.Audience,
            IssuerSigningKey = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(jwtSettings.Secret))
        };
    });
  7. Add authorization policies: 
    builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("Therapist", policy => policy.RequireRole("Therapist"));
    options.AddPolicy("Patient", policy => policy.RequireRole("Patient"));
});
  8. Register IJwtTokenGenerator in DI: services.AddScoped<IJwtTokenGenerator, JwtTokenGenerator>();
  9. Add authentication middleware in the pipeline: app.UseAuthentication(); app.UseAuthorization(); (order matters).

API Design

No endpoints – this is configuration. The generated token will be used in later tasks.

Data Flow

Client → Login Request → Auth Service → JwtTokenGenerator → Token → Client

Deliverables

  • JwtSettings class.
  • IJwtTokenGenerator interface.
  • JwtTokenGenerator implementation.
  • Authentication and authorization middleware configured in Program.cs.
  • Policies defined.

Validation Checklist

  • JWT secret is at least 32 characters.
  • Token generation produces a valid JWT when tested.
  • Authentication middleware is registered.
  • Policies are defined.

Testing Strategy

Unit Tests: Test JwtTokenGenerator with a mock user and roles – verify token contains correct claims.

Integration Tests: Not applicable – configuration is tested via endpoint calls in later tasks.

Common Pitfalls

Pitfall: Using a weak secret – must be at least 32 characters. Use Guid.NewGuid().ToString() + extra random string.
Pitfall: Forgetting to add app.UseAuthentication() before UseAuthorization() – authentication will not be run.
Tip: Store JWT secret in Azure Key Vault or environment variables – never commit to source control.

Definition of Done

  • Authentication middleware is configured.
  • JWT token generator works and is tested.
  • Policies defined.
  • Solution builds and passes tests.

Learning Resources

Must Know

  • JWT structure and claims.
  • ASP.NET Core authentication middleware.
  • Role‑based authorization.

Nice To Know

  • Refresh tokens.
  • OAuth2 / OpenID Connect.

Recommended Learning Order

  1. JWT fundamentals (2h).
  2. ASP.NET Core JWT integration (3h).
BE-AUTH-002 Auth DTOs P0 Low 2h ▾

Task Information

  • Task ID: BE-AUTH-002
  • Task Name: Auth DTOs
  • Owner: M1 (Backend Lead)
  • Dependencies: None – can be done in parallel with BE-AUTH-001
  • Complexity: Low
  • Estimated Effort: 2 hours

Objective

Create DTOs (Data Transfer Objects) for authentication operations: Login, Register, Reset Password, Token Response.

Business Purpose

Define the structure of data exchanged between frontend and backend during authentication flows.

Technical Purpose

Ensure type‑safe communication and validation of incoming requests.

Prerequisites

  • DTO concepts: Understand role of DTOs in Clean Architecture.
  • Why needed: To separate domain entities from API contracts.

Inputs

  • User entity (for mapping).

Outputs

  • LoginRequest – Email, Password.
  • RegisterRequest – Email, Password, FirstName, LastName, Role (Therapist/Patient).
  • ResetPasswordRequest – Email, Token, NewPassword.
  • TokenResponse – Token, Expiry, UserId, Email, Roles.
  • ChangePasswordRequest – CurrentPassword, NewPassword.

Detailed Backend Workflow

  1. Create LoginRequest – in Jalsa.Application/DTOs/Auth/. 
    public class LoginRequest
{
    [Required, EmailAddress]
    public string Email { get; set; }
    [Required]
    public string Password { get; set; }
}
  2. Create RegisterRequest – with similar validation.
  3. Create TokenResponse – includes token string, expiry, user details.
  4. Create ResetPasswordRequest – includes token.
  5. Create ChangePasswordRequest.
  6. Add FluentValidation validators for each DTO (if using FluentValidation) – or use DataAnnotations.

API Design

These DTOs will be used in the Auth Controller endpoints (BE-AUTH-004).

Deliverables

  • All DTO classes.
  • Validation rules (either DataAnnotations or FluentValidation).

Validation Checklist

  • DTOs have required properties and appropriate validation attributes.
  • TokenResponse includes necessary user information.

Testing Strategy

Unit Tests: Test validators (if FluentValidation) to ensure validation rules work.

Definition of Done

  • All DTOs are created and validated.
BE-AUTH-003 Auth Service P0 High 8h ▾

Task Information

  • Task ID: BE-AUTH-003
  • Task Name: Auth Service
  • Owner: M1 (Backend Lead)
  • Dependencies: BE-AUTH-001 (JWT generator), BE-AUTH-002 (DTOs), Phase 1 (UserRepository, UnitOfWork)
  • Complexity: High
  • Estimated Effort: 8 hours

Objective

Implement the authentication service with methods: RegisterAsync, LoginAsync, LogoutAsync, RefreshTokenAsync (optional), ResetPasswordAsync, ChangePasswordAsync.

Business Purpose

Handle all authentication logic – user registration, login, password management. This is the core of the security layer.

Technical Purpose

Encapsulate business logic for authentication, using repositories, password hashing, and JWT generation.

Prerequisites

  • Password hashing: BCrypt or Identity PasswordHasher.
  • Repository pattern: Access to IUserRepository and IRoleRepository.
  • JWT generator: IJwtTokenGenerator.

Inputs

  • LoginRequest, RegisterRequest, ResetPasswordRequest, ChangePasswordRequest.
  • UserRepository, RoleRepository, UnitOfWork.

Outputs

  • IAuthService interface.
  • AuthService implementation.

Detailed Backend Workflow

  1. Create IAuthService in Jalsa.Application/Interfaces/Auth/ with methods.
  2. Implement AuthService in Jalsa.Application/Services/Auth/:
    • Inject IUserRepository, IRoleRepository, IJwtTokenGenerator, IUnitOfWork, IPasswordHasher (or BCrypt).
    • RegisterAsync:
      1. Check if email already exists – throw conflict.
      2. Hash password.
      3. Create User entity with RoleId.
      4. Add via repository, save via UnitOfWork.
      5. Return user.
    • LoginAsync:
      1. Find user by email (include Role).
      2. Verify password using PasswordHasher.
      3. Generate JWT token.
      4. Return TokenResponse.
    • ResetPasswordAsync:
      1. Find user by email.
      2. Verify reset token (stored in a separate table or in User).
      3. Hash new password.
      4. Update and save.
    • ChangePasswordAsync: similar to reset but requires current password.
  3. Register service in DI: services.AddScoped<IAuthService, AuthService>();

Data Flow

Controller → AuthService → Repository → Database ↕ JwtTokenGenerator

API Design

Endpoints will be defined in BE-AUTH-004.

Deliverables

  • IAuthService interface.
  • AuthService implementation.
  • DI registration.

Validation Checklist

  • Registration creates user with hashed password.
  • Login returns valid token for correct credentials.
  • Invalid credentials throw appropriate exceptions (e.g., Unauthorized).
  • Password reset works with valid token.
  • All methods handle edge cases (user not found, email already taken).

Testing Strategy

Unit Tests: Mock repositories and test each method.

Integration Tests: Use a test database to test full flow.

Common Pitfalls

Pitfall: Not handling duplicate email during registration – check existence before inserting.
Pitfall: Using weak password hashing – use BCrypt or Identity PasswordHasher with high work factor.

Definition of Done

  • AuthService implemented and tested.
  • All methods work as expected.
BE-AUTH-004 Auth Controller P0 Medium 4h ▾

Task Information

  • Task ID: BE-AUTH-004
  • Task Name: Auth Controller
  • Owner: M1 (Backend Lead)
  • Dependencies: BE-AUTH-003 (AuthService)
  • Complexity: Medium
  • Estimated Effort: 4 hours

Objective

Create the API endpoints for authentication: /api/auth/register, /api/auth/login, /api/auth/logout, /api/auth/reset-password, /api/auth/change-password.

Business Purpose

Expose authentication functionality to the frontend, enabling users to sign up, log in, and manage passwords.

Technical Purpose

Map HTTP requests to service methods, validate input, and return appropriate HTTP responses with status codes.

Prerequisites

  • ASP.NET Core controllers, routing.
  • FluentValidation / DataAnnotations.

Inputs

  • AuthService.

Outputs

  • AuthController with endpoints.

Detailed Backend Workflow

  1. Create AuthController in Jalsa.API/Controllers/.
  2. Inject IAuthService and IMapper (if needed).
  3. Implement endpoints:
    • POST /register – accepts RegisterRequest, returns TokenResponse or 201.
    • POST /login – accepts LoginRequest, returns TokenResponse or 401.
    • POST /logout – optional, just clears client token.
    • POST /reset-password – accepts ResetPasswordRequest, returns 200.
    • POST /change-password – requires authenticated user.
  4. Add validation – use [FromBody] and model validation.
  5. Add authorization – /change-password requires [Authorize].
  6. Handle exceptions – use global exception middleware or try-catch.

Deliverables

Validation Checklist

  • Login endpoint returns token for valid credentials.
  • Password reset sends email (if implemented).

BE-AUTH-007 Password Reset – Email P1 Medium 5h ▾

Task Information

  • Task ID: BE-AUTH-007
  • Task Name: Password Reset – Email
  • Owner: M5 (DevOps / QA / AI)
  • Dependencies: BE-AUTH-003 (AuthService), Email service (EXT-EMAIL-001)
  • Complexity: Medium
  • Estimated Effort: 5 hours

Objective

Implement the password reset flow: generate a secure token, store it, and send an email with a reset link.

Business Purpose

Allow users to reset forgotten passwords via email, improving user experience and security.

Technical Purpose

Generate a time‑limited reset token (JWT or random string), store it in the User entity (or separate table), and send via email.

Prerequisites

  • Email service (SendGrid/SMTP).
  • User repository.

Inputs

  • User email.
  • Reset token generation.

Outputs

  • Password reset token stored.
  • Email sent with reset link.

Detailed Backend Workflow

  1. Add ResetToken and ResetTokenExpiry fields to User entity (or separate PasswordResetToken table).
  2. Create method GenerateResetTokenAsync – generate a cryptographically secure random string or use JWT.
  3. Store token in User (hashed, not plaintext) and set expiry (e.g., 1 hour).
  4. Send email with reset link containing token (e.g., /reset-password?token=...).
  5. Create endpoint /api/auth/forgot-password – accepts email, triggers email.
  6. Create endpoint /api/auth/reset-password – accepts token and new password, verifies token, updates password.

API Design

RouteMethodRequestResponse
/api/auth/forgot-passwordPOST{ email }200 OK
/api/auth/reset-passwordPOST{ token, newPassword }200 OK

Deliverables

  • Reset token fields in User.
  • Forgot password and reset password endpoints.
  • Email service integration.

Validation Checklist

  • Token is generated and stored.
  • Email is sent.
  • Reset works with valid token.
  • Token expires and cannot be reused.

Common Pitfalls

Pitfall: Sending the token in plaintext in the email – use HTTPS link and token as query param.

Definition of Done

  • Password reset flow works end‑to‑end.
