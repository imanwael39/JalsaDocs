🧪 Testing & QA – Phase 12

Purpose: Establish a comprehensive testing strategy to validate the entire Jalsa system – backend, frontend, AI services, and integration points. Ensure reliability, security, and performance before deployment.

📋 Tasks: 8 ⏱️ Total Effort: ~75 hours 👤 Owners: M5 (DevOps/QA Lead), M1 (Backend), M3 (Frontend) 🔗 Dependencies: All backend, frontend, and AI phases (0–11) must be complete
TST-001Unit Test Setup P1 Medium 3h ▾

Task Information

  • Task ID: TST-001
  • Task Name: Unit Test Setup
  • Owner: M5 (DevOps/QA Lead)
  • Dependencies: Phase 0 (Solution structure), all backend code
  • Complexity: Medium
  • Estimated Effort: 3 hours

Objective

Set up the testing framework across the solution: configure xUnit (or NUnit), Moq, FluentAssertions, and ensure all test projects are created and reference the correct production projects.

Business Purpose

Establish a foundation for automated testing to catch bugs early and ensure code quality.

Technical Purpose

Create test projects for each layer (Domain, Application, Infrastructure, API) and install required NuGet packages. Configure test runners and code coverage tools.

Prerequisites

Backend

  • .NET testing frameworks: xUnit, Moq, FluentAssertions.
  • Test runners: dotnet test, Visual Studio Test Explorer.

Frontend

  • None – but later will need Jasmine/Karma.

Database

  • In‑memory database for integration tests (EF Core InMemory).

AI

  • Mocking OpenAI clients for unit tests.

Inputs

  • Solution structure from Phase 0.
  • List of existing projects.

Outputs

  • Test projects: Jalsa.Domain.Tests, Jalsa.Application.Tests, Jalsa.Infrastructure.Tests, Jalsa.API.Tests.
  • All test projects reference appropriate production projects.
  • NuGet packages: xUnit, Moq, FluentAssertions, Microsoft.EntityFrameworkCore.InMemory.
  • Sample test class to verify setup.

Detailed Backend Workflow

  1. Create test projects – for each production project, create a corresponding test project (e.g., dotnet new xunit -n Jalsa.Domain.Tests).
  2. Add project references – each test project references the production project it tests (e.g., Jalsa.Domain.Tests references Jalsa.Domain).
  3. Install NuGet packages – add Moq, FluentAssertions, and InMemory EF Core to appropriate test projects.
  4. Create a base test class (optional) with common setup.
  5. Add a sample test – e.g., a simple entity test to ensure configuration works.
  6. Run dotnet test to verify all test projects build and pass.
  7. Commit test projects to source control.

Detailed Database Workflow

No database changes – configuration for in‑memory will be in integration tests.

Data Flow Diagram

dotnet test → Test Runner → Test Projects → Production Code → Results

Deliverables

  • Four test projects.
  • Sample passing test.
  • Documentation on how to write and run tests.

Validation Checklist

  • ☐ All test projects build.
  • ☐ dotnet test runs successfully.
  • ☐ Sample test passes.
  • ☐ References are correct.

Testing Strategy

This task is about setting up infrastructure; no functional tests yet.

Common Pitfalls

Pitfall: Not referencing the correct projects – ensure each test project has a project reference to the correct production project.

Definition of Done

  • Test projects created, build, and pass a sample test.

Learning Resources

Must Know: xUnit basics, Moq, FluentAssertions.

TST-002Backend Unit Tests P1 High 16h ▾

Task Information

  • Task ID: TST-002
  • Task Name: Backend Unit Tests
  • Owner: M1 (Backend Lead) + M2 (Backend)
  • Dependencies: TST-001, all backend tasks (DB-*, BE-*)
  • Complexity: High
  • Estimated Effort: 16 hours

Objective

Write unit tests for all backend services, controllers, repositories, and validators. Aim for high code coverage (>=80%).

Business Purpose

Ensure that core business logic (authentication, patient management, sessions, exercises, reports, AI services) behaves correctly in isolation.

Technical Purpose

Test each unit of code independently, using mocks for dependencies (repositories, external APIs). Validate error handling, edge cases, and business rules.

Prerequisites

  • Test setup from TST-001.
  • Moq for mocking dependencies.
  • FluentAssertions for readable assertions.

Inputs

  • All backend source code: entities, DTOs, services, controllers, repositories, validators.

Outputs

  • Unit test classes for each service, controller, and validator.
  • Test coverage report showing >=80% coverage.
  • All tests passing.

Detailed Backend Workflow

  1. Prioritise critical modules – Auth, Patient, Session, Exercise, Reports, AI (with mocks).
  2. For each service:
    • Create a test class (e.g., AuthServiceTests).
    • Mock all injected dependencies (repositories, unit of work, hasher, token generator).
    • Write tests for each public method: success, failure (e.g., user not found, duplicate email, invalid credentials).
  3. For each controller:
    • Mock the service layer.
    • Test each endpoint: correct status codes, DTO mapping, error responses.
  4. For validators: test validation rules with valid and invalid data.
  5. For repositories (optional): test custom query methods with in‑memory database.
  6. Run tests and ensure all pass. Use code coverage tools (Coverlet) to measure coverage.
  7. Fix any failing tests and improve coverage as needed.

Data Flow Diagram

Test Class → Mocks → Service Under Test → Assertions

Deliverables

  • Unit test classes for all backend modules.
  • Coverage report (HTML or XML).

Validation Checklist

  • ☐ All test classes exist.
  • ☐ Each public method has at least one success and one failure test.
  • ☐ Coverage >=80%.
  • ☐ All tests pass.

Testing Strategy

Unit Tests: Complete as described.

Common Pitfalls

Pitfall: Over‑mocking – only mock external dependencies; use real implementations for simple value objects.
Pitfall: Not testing edge cases – e.g., null inputs, empty collections, invalid enums.

Definition of Done

  • All backend unit tests pass with >=80% coverage.

Learning Resources

Must Know: Moq, FluentAssertions, xUnit theory/inline data.

TST-003Integration Tests – API P1 High 12h ▾

Task Information

  • Task ID: TST-003
  • Task Name: Integration Tests – API
  • Owner: M5 (DevOps/QA) with M1
  • Dependencies: TST-001, all backend APIs
  • Complexity: High
  • Estimated Effort: 12 hours

Objective

Write integration tests for the entire API layer using TestServer (ASP.NET Core) and an in‑memory database to test end‑to‑end scenarios including database interactions, authentication, and validation.

Business Purpose

Ensure that the API endpoints work correctly when integrated with the database and external dependencies (mocked). Validate the full request/response cycle.

Technical Purpose

Test the complete stack: controller → service → repository → database (in‑memory) → response. Validate HTTP status codes, error messages, and DTO mappings.

Prerequisites

  • EF Core InMemory provider.
  • TestServer from Microsoft.AspNetCore.TestHost.
  • Startup class configured for testing.

Inputs

  • All API controllers.
  • Application DB context.

Outputs

  • Integration test classes for key API flows: auth, patient CRUD, sessions, exercises, reports, chat, crisis alerts.
  • Tests pass.

Detailed Backend Workflow

  1. Set up a test fixture – create a custom WebApplicationFactory that uses InMemory database and replaces external services with mocks or test doubles.
  2. Write test methods for each critical endpoint:
    • POST /api/auth/login – valid and invalid credentials.
    • GET /api/patients – with authentication.
    • POST /api/patients – create patient.
    • PUT /api/patients/{id} – update.
    • DELETE /api/patients/{id} – archive.
    • Session endpoints, exercise endpoints, report generation, chat send, etc.
  3. Use HttpClient from the test server to send requests and assert responses.
  4. Seed test data in the in‑memory database for consistent tests.
  5. Run tests and ensure they pass.

Data Flow Diagram

Test → HttpClient → TestServer → Middleware → Controller → Service/Repository → InMemory DB → Response

Deliverables

  • Integration test classes for all major API flows.
  • Test data seeds.

Validation Checklist

  • ☐ All critical endpoints have integration tests.
  • ☐ Tests pass in CI pipeline.
  • ☐ Tests cover success and failure cases.
  • ☐ Database state is reset between tests.

Testing Strategy

Integration Tests: Use TestServer and InMemory DB.

Common Pitfalls

Pitfall: InMemory provider may not support all SQL features – use SQLite InMemory for better compatibility if needed.
Pitfall: Not resetting database between tests – ensure each test is isolated.

Definition of Done

  • Integration tests pass and provide coverage for key flows.

Learning Resources

Must Know: ASP.NET Core TestServer, EF Core InMemory, xUnit fixtures.

TST-004Frontend Unit Tests P2 Medium 12h ▾

Task Information

  • Task ID: TST-004
  • Task Name: Frontend Unit Tests
  • Owner: M3 (Frontend Lead) + M4 (Frontend)
  • Dependencies: All frontend code (FE-*)
  • Complexity: Medium
  • Estimated Effort: 12 hours

Objective

Write unit tests for Angular components, services, guards, and pipes using Jasmine and Karma.

Business Purpose

Ensure the frontend logic (routing, state management, form validation, service calls) works correctly and is maintainable.

Technical Purpose

Test Angular components in isolation, mock dependencies, and verify UI state changes.

Prerequisites

  • Jasmine, Karma configured (Angular default).
  • Angular TestBed for component testing.

Inputs

  • All Angular components, services, guards, pipes.

Outputs

  • Unit test files (.spec.ts) for each component/service/guard.
  • Tests pass.
  • Coverage report.

Detailed Frontend Workflow

  1. For each component:
    • Create a test suite using TestBed.
    • Mock services (using jasmine.createSpyObj).
    • Test component creation, DOM rendering, event handling, and output emissions.
  2. For services: test HTTP calls with HttpClientTestingModule.
  3. For guards: test canActivate logic with mock AuthService.
  4. For pipes: test transformation logic.
  5. Run ng test and ensure all tests pass.
  6. Check coverage with ng test --code-coverage and aim for >=70%.

Data Flow Diagram

Test → TestBed → Component → Mocks → Assertions

Deliverables

  • Unit test files for frontend.
  • Passing tests.

Validation Checklist

  • ☐ All components have basic test coverage.
  • ☐ Services have tests for HTTP methods.
  • ☐ Guards tested.
  • ☐ Tests pass.

Testing Strategy

Unit Tests: Angular Jasmine/Karma.

Common Pitfalls

Pitfall: Not mocking HTTP requests – use HttpClientTestingController.

Definition of Done

  • Frontend unit tests pass.

Learning Resources

Must Know: Angular TestBed, Jasmine spies, HttpClientTestingModule.

TST-005E2E Tests (Cypress) P2 Medium 16h ▾

Task Information

  • Task ID: TST-005
  • Task Name: E2E Tests (Cypress)
  • Owner: M5 (DevOps/QA) with M3
  • Dependencies: All frontend and backend (deployed in test environment)
  • Complexity: Medium
  • Estimated Effort: 16 hours

Objective

Write end‑to‑end tests using Cypress to cover critical user journeys – login, patient management, session recording, exercise tracking, report generation, chat, and crisis alerts.

Business Purpose

Ensure that the system works as a whole from the user's perspective, verifying UI interactions, API calls, and database updates.

Technical Purpose

Test the full stack in a production‑like environment with a real database (or test database). Validate that frontend and backend integrate correctly.

Prerequisites

  • Cypress installed.
  • A test environment with the application running (e.g., localhost with Docker Compose).
  • Seed data for consistent tests.

Inputs

  • Frontend application, backend API.

Outputs

  • Cypress test files (.cy.ts) for key flows.
  • Tests pass in CI.

Detailed Frontend Workflow

  1. Set up Cypress – ng add @cypress/schematic or manual install.
  2. Create test specs:
    • Login (therapist and patient).
    • Patient CRUD (create, edit, archive).
    • Create session note with voice memo (if possible).
    • Exercise assignment and patient status update.
    • Generate and approve referral report.
    • Chat – start session, send message, AI response.
    • Crisis alert – manual trigger.
  3. Use custom commands for login and seeding.
  4. Run Cypress in headless mode for CI.
  5. Ensure tests are deterministic – clean database state.

Data Flow Diagram

Cypress → Browser → Frontend → API → Backend → Database

Deliverables

  • Cypress test files.
  • Test reports.

Validation Checklist

  • ☐ All critical flows have E2E tests.
  • ☐ Tests pass in CI.
  • ☐ Tests cover login, patient, session, exercise, report, chat.

Testing Strategy

E2E Tests: Cypress.

Common Pitfalls

Pitfall: Flaky tests due to timing – use cy.intercept to wait for API calls.

Definition of Done

  • E2E tests pass.

Learning Resources

Must Know: Cypress API, custom commands, fixtures.

TST-006Performance / Load Testing P2 Medium 8h ▾

Task Information

  • Task ID: TST-006
  • Task Name: Performance / Load Testing
  • Owner: M5 (DevOps/QA)
  • Dependencies: All backend APIs, database
  • Complexity: Medium
  • Estimated Effort: 8 hours

Objective

Perform load testing on critical API endpoints (e.g., patient search, session retrieval, report generation) using tools like k6 or JMeter to ensure they can handle expected concurrency.

Business Purpose

Validate that the system meets performance requirements (response time, throughput) under load, ensuring a smooth user experience.

Technical Purpose

Identify bottlenecks in the API, database, or AI services. Establish baseline performance metrics.

Prerequisites

  • Load testing tool (k6 recommended).
  • A staging environment with representative data size.

Inputs

  • API endpoints to test: GET /api/patients (search), GET /api/sessions, POST /api/reports/generate, POST /api/chat/send.

Outputs

  • Load test scripts.
  • Performance reports (response times, error rates, throughput).
  • Identified bottlenecks and recommendations.

Detailed Backend Workflow

  1. Install k6 or JMeter.
  2. Write test scripts for each critical endpoint:
    • Simulate concurrent users (e.g., 50, 100, 200).
    • Include authentication (obtain token).
    • Measure response time (p95, p99).
  3. Run tests against staging environment.
  4. Analyze results – if response times exceed acceptable thresholds (e.g., <500ms for non‑AI, <3s for AI), investigate and optimize.
  5. Document findings and propose improvements (indexes, caching, query optimizations).

Data Flow Diagram

k6 → API → Database → Response → Metrics

Deliverables

  • Load test scripts.
  • Performance report.

Validation Checklist

  • ☐ Tests execute successfully.
  • ☐ Response times are within acceptable limits.
  • ☐ Bottlenecks identified.

Testing Strategy

Load Tests: k6.

Common Pitfalls

Pitfall: Running tests on a local environment with limited resources – use a representative environment.

Definition of Done

  • Performance report delivered.

Learning Resources

Must Know: k6 scripting, load test concepts.

TST-007Security Testing P1 Medium 6h ▾

Task Information

  • Task ID: TST-007
  • Task Name: Security Testing
  • Owner: M5 (DevOps/QA) with M1
  • Dependencies: All backend APIs
  • Complexity: Medium
  • Estimated Effort: 6 hours

Objective

Perform security testing using OWASP ZAP to scan for common vulnerabilities (SQL injection, XSS, CSRF, insecure headers, JWT vulnerabilities).

Business Purpose

Ensure the application is secure and protects patient data (PHI) in compliance with healthcare regulations.

Technical Purpose

Identify security misconfigurations, weak authentication, and data exposure risks.

Prerequisites

  • OWASP ZAP (or similar tool).
  • Application running in a test environment.

Inputs

  • Base URL of the API.

Outputs

  • Security scan report.
  • List of vulnerabilities and remediation steps.

Detailed Backend Workflow

  1. Set up ZAP – run as a daemon or use the desktop UI.
  2. Perform automated scan – spider the API endpoints (with authentication token).
  3. Analyze results – review alerts (High, Medium, Low).
  4. Fix critical issues – e.g., missing security headers, weak encryption, exposed error details.
  5. Re‑scan to verify fixes.
  6. Document remaining accepted risks.

Data Flow Diagram

ZAP → API → Vulnerability Report

Deliverables

  • Security scan report.
  • Remediation actions taken.

Validation Checklist

  • ☐ Scan completed without critical alerts.
  • ☐ All high/medium issues addressed.
  • ☐ JWT token handling is secure (HTTP Only, secure flag).

Testing Strategy

Security Tests: OWASP ZAP.

Common Pitfalls

Pitfall: Not including authentication token in ZAP – ensure session is properly configured.

Definition of Done

  • Security scan passes with no critical vulnerabilities.

Learning Resources

Must Know: OWASP Top 10, ZAP usage.

TST-008User Acceptance Testing P0 Medium 8h ▾

Task Information

  • Task ID: TST-008
  • Task Name: User Acceptance Testing
  • Owner: M5 (DevOps/QA) – facilitated by Product Owner
  • Dependencies: All previous phases complete, system deployed to UAT environment
  • Complexity: Medium
  • Estimated Effort: 8 hours

Objective

Conduct User Acceptance Testing (UAT) with stakeholders (therapists, patients) to validate that the system meets business requirements and is ready for production.

Business Purpose

Ensure the software is fit for purpose and gains stakeholder approval before going live.

Technical Purpose

Provide a formal sign‑off process, demonstrating that all features work as expected in a production‑like environment.

Prerequisites

  • UAT environment deployed.
  • Test data seeded.
  • UAT test cases prepared (based on acceptance criteria).

Inputs

  • UAT test cases.
  • Stakeholders (therapists, patients).

Outputs

  • UAT sign‑off document.
  • List of any issues found and resolved.

Detailed Backend Workflow

  1. Prepare UAT environment – deploy latest build, run migrations, seed data.
  2. Prepare UAT test scripts – cover all user journeys (therapist and patient).
  3. Run UAT sessions – stakeholders follow test scripts and document issues.
  4. Fix any critical issues found.
  5. Obtain sign‑off from stakeholders.

Data Flow Diagram

Stakeholder → UAT System → Feedback → Fixes → Sign‑off

Deliverables

  • UAT test execution report.
  • Sign‑off document.

Validation Checklist

  • ☐ All UAT test cases executed.
  • ☐ All blockers resolved.
  • ☐ Sign‑off obtained.

Testing Strategy

Acceptance Tests: Manual testing by stakeholders.

Common Pitfalls

Pitfall: Not having clear test cases – ensure they are detailed and traceable to requirements.

Definition of Done

  • UAT signed off.

Learning Resources

Must Know: UAT process, test case management.
