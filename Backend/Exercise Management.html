🏋️ Exercise Management – Phase 5

Purpose: Enable therapists to assign exercises to patients, track completion status, allow patients to submit reflections, and send automated reminders for upcoming due dates. This phase builds the backend and frontend for exercise management.

📋 Tasks: 13 (8 backend + 5 frontend) ⏱️ Total Effort: ~54 hours 👤 Owners: M2 (Backend) + M4 (Frontend) 🔗 Dependencies: Phase 3 (Patient Management) complete
BE-EXE-001Exercise DTOs P0 Low 3h ▾

Task Information

  • Task ID: BE-EXE-001
  • Task Name: Exercise DTOs
  • Owner: M2 (Backend)
  • Dependencies: Phase 1 – Exercise and PatientExercise entities exist
  • Complexity: Low
  • Estimated Effort: 3 hours

Objective

Create DTOs for exercise management: ExerciseCreateDto, ExerciseUpdateDto, ExerciseViewDto, AssignDto, CompletionDto.

Business Purpose

Define data contracts for assigning exercises, updating details, and tracking completion.

Technical Purpose

Provide type‑safe communication between frontend and backend, with validation.

Prerequisites

  • Data Annotations / FluentValidation: For validation.
  • DTO pattern: Understand role in Clean Architecture.

Inputs

  • Exercise and PatientExercise entities from Phase 1.

Outputs

  • ExerciseCreateDto – Title, Description, Type, DefaultDueDays.
  • ExerciseUpdateDto – Id, Title, Description, Type, DefaultDueDays.
  • ExerciseViewDto – Id, Title, Description, Type, DefaultDueDays, CreatedAt.
  • AssignDto – ExerciseId, PatientId, DueDate (optional, can use default).
  • CompletionDto – PatientExerciseId, Status (Complete/Partial/Skipped), Reflection (string).

Detailed Backend Workflow

  1. Create DTO classes in Jalsa.Application/DTOs/Exercise/.
  2. Add validation attributes: Title required, MaxLength, Status enum validation.
  3. For AssignDto, validate that DueDate is not in the past.

Deliverables

  • Five DTO classes with validation.

Validation Checklist

  • Required fields marked.
  • String lengths constrained.
  • Date validation.

Testing Strategy

Unit Tests: Validate DTOs with DataAnnotations or FluentValidation.

Definition of Done

  • DTOs created and committed.

Learning Resources

Must Know: DataAnnotations, enum usage.

BE-EXE-006Exercise Reminder – Background P1 Medium 4h ▾

Task Information

  • Task ID: BE-EXE-006
  • Task Name: Exercise Reminder – Background
  • Owner: M5 (DevOps/QA/AI)
  • Dependencies: BE-EXE-003 (ExerciseService), Notification service (EXT-NOT-001)
  • Complexity: Medium
  • Estimated Effort: 4 hours

Objective

Implement a recurring background job (Hangfire) that checks for exercises due soon (e.g., in 48 hours) and sends reminders to patients.

Business Purpose

Help patients remember to complete their exercises.

Technical Purpose

Use Hangfire to run a daily job that queries PatientExercise records with DueDate in range and status not Complete, then sends notifications.

Prerequisites

  • Hangfire setup.
  • Notification service (email/SMS).
  • PatientExercise repository.

Inputs

  • Database of PatientExercise.

Outputs

  • Reminder notifications sent.

Detailed Backend Workflow

  1. Create a Hangfire recurring job (daily at 9:00 AM).
  2. Query PatientExercise where DueDate <= now + 48h and DueDate > now, Status != Complete.
  3. Group by patient, prepare reminder message.
  4. Call INotificationService.SendReminderAsync for each patient (email/SMS).
  5. Log reminders sent.

Data Flow

Hangfire → Query → Notification Service → Patient

Deliverables

  • Hangfire job registration.
  • Reminder logic.

Validation Checklist

  • Job runs on schedule.
  • Only pending exercises selected.
  • Notifications sent.
  • No duplicate reminders.

Common Pitfalls

Pitfall: Sending duplicate reminders. Ensure job checks last reminder sent timestamp or only sends once per exercise.

Definition of Done

  • Reminder job functional.
