👤 Phase 3 – Patient Management

Purpose: Build the patient management backend – CRUD, search, archive, digital intake forms, OCR integration for image‑based intake, and psychological assessment recording. This phase establishes the central data management for patients.

📋 Tasks: 10 ⏱️ Total Effort: ~48 hours 👤 Owners: M2 (Backend) + M5 (AI/OCR) 🔗 Dependencies: Phase 2 (Authentication) must be complete
BE-PAT-001 Patient DTOs P0 Low 3h ▾

Task Information

  • Task ID: BE-PAT-001
  • Task Name: Patient DTOs
  • Owner: M2 (Backend)
  • Dependencies: Phase 1 – Patient entity exists
  • Complexity: Low
  • Estimated Effort: 3 hours

Objective

Create DTOs for patient management: PatientCreateDto, PatientUpdateDto, PatientViewDto, PatientSearchDto.

Business Purpose

Define the structure of data exchanged between frontend and backend for patient CRUD operations.

Technical Purpose

Ensure type‑safe communication and data validation. Separate domain entities from API contracts.

Prerequisites

  • DTO pattern: Understand role of DTOs in Clean Architecture.
  • Data annotations: For validation.

Inputs

  • PatientProfile entity from Phase 1.

Outputs

  • PatientCreateDto – fields: FirstName, LastName, Email, DateOfBirth, Gender, EmergencyContact, Notes.
  • PatientUpdateDto – similar to Create but with Id.
  • PatientViewDto – includes Id, full name, age, etc.
  • PatientSearchDto – search criteria (name, email, date range).

Detailed Backend Workflow

  1. Create PatientCreateDto with required properties and validation attributes (e.g., [Required], [EmailAddress]).
  2. Create PatientUpdateDto – include an Id field.
  3. Create PatientViewDto – include computed fields like Age.
  4. Create PatientSearchDto – with optional filters.
  5. Place all in Jalsa.Application/DTOs/Patient/.

API Design

These DTOs will be used in the Patient Controller (BE-PAT-004).

Deliverables

  • Four DTO classes with validation.

Validation Checklist

  • All required fields have [Required].
  • Email is validated with [EmailAddress].
  • Dates are DateTime (or DateOnly).

Testing Strategy

Unit Tests: Test validation attributes using Validator.TryValidateObject.

Definition of Done

  • DTOs created and validated.

Learning Resources

  • DataAnnotations in .NET.
BE-PAT-002 Patient Repository P0 Medium 5h ▾

Task Information

  • Task ID: BE-PAT-002
  • Task Name: Patient Repository
  • Owner: M2 (Backend)
  • Dependencies: DB-011 (Generic Repository), DB-002 (Patient entity)
  • Complexity: Medium
  • Estimated Effort: 5 hours

Objective

Implement the patient repository with custom methods for search, paging, archive, and including related data.

Business Purpose

Provide data access for patient CRUD, search, and archive operations with efficient querying.

Technical Purpose

Extend the generic repository with patient‑specific queries.

Prerequisites

  • Generic Repository: IGenericRepository<PatientProfile>.
  • EF Core: Include, Where, OrderBy, Skip/Take.

Inputs

  • DbContext, Patient entity.

Outputs

  • IPatientRepository interface.
  • PatientRepository implementation.
  • DI registration.

Detailed Backend Workflow

  1. Create IPatientRepository extending IGenericRepository<PatientProfile>.
  2. Add custom methods:
    • GetPagedAsync(int page, int pageSize, string searchTerm, bool includeArchived = false)
    • SearchAsync(string searchTerm)
    • ArchiveAsync(Guid patientId)
    • GetWithDetailsAsync(Guid patientId) – include IntakeForms, Assessments, Sessions.
    • GetByUserIdAsync(Guid userId) – to get patient profile from user.
  3. Implement PatientRepository:
    • Inject DbContext.
    • Implement methods using DbSet.
    • Use Include for navigation properties.
    • Use Where to filter by IsDeleted (soft‑delete).
    • Use OrderBy and Skip/Take for paging.
  4. Register in DI: services.AddScoped<IPatientRepository, PatientRepository>();

Data Flow

Service → IPatientRepository → DbContext → Database

Deliverables

  • IPatientRepository interface.
  • PatientRepository implementation.
  • Unit tests for repository.

Validation Checklist

  • GetPagedAsync returns correct page with sorting.
  • Search returns matching patients.
  • Archive sets IsDeleted = true.
  • GetWithDetailsAsync includes all related data.

Common Pitfalls

Pitfall: Forgetting to add Include for navigation properties – leads to N+1 queries. Use eager loading.

Definition of Done

  • Repository implemented and tested.
BE-PAT-003 Patient Service P0 Medium-High 6h ▾

Task Information

  • Task ID: BE-PAT-003
  • Task Name: Patient Service
  • Owner: M2 (Backend)
  • Dependencies: BE-PAT-001 (DTOs), BE-PAT-002 (Repository), AutoMapper
  • Complexity: Medium-High
  • Estimated Effort: 6 hours

Objective

Implement the patient service layer with CRUD operations, search, archive, and mapping between DTOs and entities.

Business Purpose

Encapsulate business logic for patient management – create, read, update, delete (archive), search.

Technical Purpose

Coordinate repository operations, handle validation, and map DTOs using AutoMapper.

Prerequisites

  • AutoMapper: For mapping between DTO and entity.
  • FluentValidation (or manual validation).
  • UnitOfWork for transaction management.

Inputs

  • IPatientRepository, IMapper, IUnitOfWork.

Outputs

  • IPatientService interface.
  • PatientService implementation.

Detailed Backend Workflow

  1. Create IPatientService with methods:
    • Task<PatientViewDto> GetByIdAsync(Guid id)
    • Task<PagedResult<PatientViewDto>> GetPagedAsync(int page, int pageSize, string searchTerm)
    • Task<PatientViewDto> CreateAsync(PatientCreateDto dto)
    • Task<PatientViewDto> UpdateAsync(PatientUpdateDto dto)
    • Task ArchiveAsync(Guid id)
    • Task RestoreAsync(Guid id)
  2. Implement PatientService:
    • Inject IPatientRepository, IMapper, IUnitOfWork.
    • Create method: map DTO to entity, set audit fields (CreatedAt), add, save.
    • Update: get entity, map changes, set UpdatedAt, save.
    • Archive: set IsDeleted = true, save.
    • Get methods: use repository, map to ViewDto.
  3. Configure AutoMapper profiles – map PatientCreateDto → PatientProfile, PatientProfile → PatientViewDto.
  4. Register in DI: services.AddScoped<IPatientService, PatientService>();

Data Flow

Controller → IPatientService → Repository → Database

Deliverables

  • IPatientService interface.
  • PatientService implementation.
  • AutoMapper profiles.

Validation Checklist

  • Create succeeds and returns ViewDto.
  • Update modifies entity correctly.
  • Archive sets IsDeleted.
  • Search returns filtered results.
  • Validation errors are thrown appropriately.

Testing Strategy

Unit Tests: Mock repository and test each service method.

Definition of Done

  • Service implemented and unit tested.
BE-PAT-004 Patient Controller P0 Medium 4h ▾

Task Information

  • Task ID: BE-PAT-004
  • Task Name: Patient Controller
  • Owner: M2 (Backend)
  • Dependencies: BE-PAT-003 (PatientService)
  • Complexity: Medium
  • Estimated Effort: 4 hours

Objective

Create the API endpoints for patient management: CRUD, search, archive, restore.

Business Purpose

Expose patient management functionality to the frontend.

Technical Purpose

Map HTTP requests to service methods, handle validation, and return appropriate responses.

Prerequisites

  • ASP.NET Core controllers.
  • Authorization (role‑based).

Inputs

  • IPatientService.

Outputs

  • PatientController with endpoints.

Detailed Backend Workflow

  1. Create PatientController with route [Route("api/patients")].
  2. Inject IPatientService.
  3. Implement endpoints:
    • GET /{id} – get patient by ID.
    • GET – search with pagination (query params: page, pageSize, searchTerm).
    • POST – create patient.
    • PUT /{id} – update patient.
    • DELETE /{id} – archive (soft delete).
    • POST /{id}/restore – restore archived.
  4. Apply authorization – [Authorize(Roles = "Therapist")].
  5. Handle exceptions – use global middleware.

API Design

/api/patientsGETQuery: page, pageSize, searchTermPagedResult<PatientViewDto>Therapist
/api/patientsPOSTPatientCreateDtoPatientViewDto (201)Therapist
/api/patients/{id}PUTPatientUpdateDtoPatientViewDtoTherapist
/api/patients/{id}DELETENone204 No ContentTherapist
/api/patients/{id}/restorePOSTNone200 OKTherapist

Data Flow

Client → PatientController → PatientService → Repository → Database ↕ Response

Deliverables

  • PatientController with all endpoints.

Validation Checklist

  • GET returns correct patient.
  • POST creates patient with 201.
  • PUT updates correctly.
  • DELETE archives (soft delete).
  • Search works with filters.
  • Authorization works – only therapists.

Testing Strategy

Integration Tests: Test all endpoints with test database.

Definition of Done

BE-PAT-006 Intake Form – Image Upload P1 Medium 4h ▾

Task Information

  • Task ID: BE-PAT-006
  • Task Name: Intake Form – Image Upload
  • Owner: M5 (DevOps / AI)
  • Dependencies: BE-PAT-005, Blob storage service
  • Complexity: Medium
  • Estimated Effort: 4 hours

Objective

Allow uploading an image of a paper intake form. Store the image in blob storage and save the URL in the IntakeForm entity.

Business Purpose

Facilitate digitisation of paper intake forms by uploading images.

Technical Purpose

Integrate with Azure Blob / AWS S3 to store images, and update the IntakeForm entity with the image URL.

Prerequisites

  • Blob Storage SDK: Azure.Storage.Blobs or AWS S3.
  • File handling: IFormFile.

Inputs

  • Patient ID, image file.

Outputs

  • Image URL stored in IntakeForm.

Detailed Backend Workflow

  1. Create endpoint POST /api/patients/{patientId}/intake/image.
  2. Accept IFormFile (multipart/form-data).
  3. Validate file – type (image/jpeg, image/png), size (max 5MB).
  4. Upload to blob – generate unique filename (e.g., intake_{patientId}_{timestamp}.jpg).
  5. Get URL and update the IntakeForm entity (create if not exists).
  6. Return the URL in response.

API Design

RouteMethodRequestResponse
/api/patients/{patientId}/intake/imagePOSTmultipart/form-data with file{ imageUrl }

Deliverables

  • Image upload endpoint.
  • Blob storage integration.

Validation Checklist

  • File type and size validation works.
  • Image is uploaded and URL is stored.

Definition of Done

  • Image upload works.
BE-PAT-007 Intake Form – OCR Integration P1 High 6h ▾

Task Information

  • Task ID: BE-PAT-007
  • Task Name: Intake Form – OCR Integration
  • Owner: M5 (DevOps / AI)
  • Dependencies: BE-PAT-006 (image upload), BE-AI-001 (Vision OCR)
  • Complexity: High
  • Estimated Effort: 6 hours

Objective

Integrate Vision OCR (GPT‑4o with Vision or Azure Form Recognizer) to extract structured data from the uploaded intake form image. Populate the IntakeForm entity's ExtractedData field with the parsed information.

Business Purpose

Automate data entry from paper intake forms, reducing manual effort and errors.

Technical Purpose

Call the Vision OCR API with the image URL, parse the response, and store the extracted data as JSON.

Prerequisites

  • Vision OCR service: OpenAI GPT‑4o with Vision or Azure Form Recognizer.
  • Prompt engineering: To structure the output.

Inputs

  • Image URL (or image bytes).

Outputs

  • ExtractedData JSON in IntakeForm.

Detailed Backend Workflow

  1. Create endpoint POST /api/intake/{intakeFormId}/ocr or trigger automatically after image upload.
  2. Retrieve image URL from IntakeForm.
  3. Call Vision OCR – using OpenAI Vision API with a prompt like "Extract the following fields from this intake form: name, date of birth, reason for visit, etc. Return as JSON."
  4. Parse response – validate the JSON structure.
  5. Update IntakeForm – set ExtractedData to the JSON string.
  6. Return the extracted data.

Data Flow

Image URL → Vision OCR API → Extracted JSON → IntakeForm.ExtractedData

Deliverables

  • OCR trigger endpoint.
  • Extraction logic.

Validation Checklist

  • OCR returns structured data.
  • Data is stored correctly.
  • Fallback for failed OCR (manual entry).

Common Pitfalls

Pitfall: Poor image quality – implement preprocessing or use a higher‑quality model.

Definition of Done

  • OCR integration works.

BE-PAT-009 Patient Validation P1 Medium 3h ▾

Task Information

  • Task ID: BE-PAT-009
  • Task Name: Patient Validation
  • Owner: M2 (Backend)
  • Dependencies: BE-PAT-001 (DTOs)
  • Complexity: Medium
  • Estimated Effort: 3 hours

Objective

Implement FluentValidation validators for all patient DTOs to ensure data integrity.

Business Purpose

Prevent invalid data from entering the system (e.g., empty names, invalid emails).

Technical Purpose

Centralise validation logic using FluentValidation.

Prerequisites

  • FluentValidation: Understanding of rules, validators.

Inputs

  • Patient DTOs.

Outputs

  • PatientCreateDtoValidator, PatientUpdateDtoValidator.

Detailed Backend Workflow

  1. Install FluentValidation and FluentValidation.AspNetCore.
  2. Create validator classes in Jalsa.Application/Validators/Patient/.
  3. Define rules – RuleFor(x => x.FirstName).NotEmpty().MaximumLength(100); etc.
  4. Register validators in DI: services.AddValidatorsFromAssemblyContaining<PatientCreateDtoValidator>();
  5. Integrate – use ValidationFilter or manual validation in service.

Deliverables

  • Validator classes.
  • DI registration.

Validation Checklist

  • All DTOs have validators.
  • Rules cover required fields, lengths, formats.
