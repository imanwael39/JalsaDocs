🚀 Jalsa – Deployment & DevOps Phase Implementation Playbook Phase 13 · 11 Tasks · Production Readiness

🚀 Deployment & DevOps – Phase 13

Purpose: Prepare the Jalsa system for production deployment. This phase covers infrastructure as code, CI/CD pipeline automation, containerization, monitoring, logging, secrets management, disaster recovery, and operational runbooks. It ensures the system is reliable, scalable, and maintainable in production.

📋 Tasks: 11 ⏱️ Total Effort: ~72 hours 👤 Owners: M5 (DevOps Lead), M1 (Backend), M3 (Frontend) 🔗 Dependencies: All previous phases (0–12) must be complete
DEV-001Infrastructure as Code P1 High 10h ▾

Task Information

  • Task ID: DEV-001
  • Task Name: Infrastructure as Code
  • Owner: M5 (DevOps Lead)
  • Dependencies: Cloud provider access (Azure/AWS/GCP), architecture decisions from earlier phases
  • Complexity: High
  • Estimated Effort: 10 hours

Objective

Define all cloud infrastructure resources using Infrastructure as Code (IaC) – Terraform or Bicep – to provision the database, blob storage, cache, vector store, and networking components in a repeatable and version‑controlled manner.

Business Purpose

Enable consistent, auditable, and automated provisioning of cloud resources, reducing human error and accelerating environment creation (dev, staging, prod).

Technical Purpose

Ensure all infrastructure (PostgreSQL, Redis, Blob Storage, Vector Store, Load Balancer, etc.) is defined as code and can be deployed with a single command, supporting disaster recovery and scaling.

Prerequisites

Backend

  • Terraform / Bicep: Understanding of resource declaration, state management, modules.
  • Cloud provider SDK: Azure/AWS CLI or equivalent.

Frontend

  • None – but frontend deployment will use the same infrastructure.

Database

  • PostgreSQL instance with pgvector extension.
  • Redis cache.

AI

  • OpenAI API access (keys stored in secrets).

Inputs

  • Cloud subscription/account.
  • Architecture decisions (database size, storage account, network configuration).

Outputs

  • Terraform/Bicep scripts for all resources.
  • Variable files for dev/staging/prod.
  • State management (remote backend).
  • Provisioned resources in at least one environment (staging).

Detailed Backend Workflow

  1. Set up Terraform/Bicep – install tools, configure authentication.
  2. Define resource modules:
    • Resource Group / Project.
    • PostgreSQL server with pgvector extension (or use Azure Database for PostgreSQL flexible).
    • Redis cache (standard tier).
    • Blob storage account (for images, audio, PDFs).
    • Vector store – if using pgvector, it's part of PostgreSQL; otherwise, separate service.
    • Network: VNet, subnets, security groups.
    • Load balancer / Application Gateway (if needed).
    • Kubernetes cluster or App Service for API and Angular.
  3. Store state remotely – use Azure Storage or S3 for Terraform state.
  4. Parameterise – create separate .tfvars files for dev, staging, prod.
  5. Run terraform plan and validate.
  6. Apply to staging environment – verify resources are created.
  7. Commit scripts to repository.

Detailed Database Workflow

  • Provision PostgreSQL with pgvector extension enabled (see DB-014).
  • Configure firewall rules to allow access from API and CI/CD.

Data Flow

terraform apply → Cloud Provider API → Resources → State File

Deliverables

  • IaC scripts.
  • Remote state configuration.
  • Provisioned staging environment.

Validation Checklist

  • ☐ All resources are defined in code.
  • ☐ Terraform plan shows no unexpected changes.
  • ☐ Staging resources are created and accessible.
  • ☐ Secrets are not hard‑coded (use variables).

Testing Strategy

Integration: Apply to a test environment and verify connectivity.

Common Pitfalls

Pitfall: Not storing state remotely – leads to conflicts in CI. Always use remote state.
Pitfall: Hard‑coding credentials – use environment variables or Azure Key Vault.

Definition of Done

  • IaC scripts committed and staging environment provisioned.

Learning Resources

Must Know: Terraform syntax, provider configuration.

Nice To Know: Bicep, ARM templates.

DEV-002CI/CD Pipeline (Full) P0 High 8h ▾

Task Information

  • Task ID: DEV-002
  • Task Name: CI/CD Pipeline (Full)
  • Owner: M5 (DevOps Lead)
  • Dependencies: DEV-001 (IaC), FND-004 (basic CI)
  • Complexity: High
  • Estimated Effort: 8 hours

Objective

Extend the basic CI pipeline to a full CI/CD pipeline that builds, tests, and deploys the backend API and frontend Angular application to dev, staging, and production environments on every push to respective branches (develop → staging, main → production).

Business Purpose

Automate the deployment process to reduce manual errors, accelerate release cycles, and ensure consistent deployments across environments.

Technical Purpose

Integrate build, test (unit, integration, E2E), containerisation, and deployment steps. Use Docker images, push to container registry, and deploy to Kubernetes or App Service.

Prerequisites

  • CI platform: GitHub Actions / Azure DevOps.
  • Container registry (ACR/Docker Hub).
  • Kubernetes cluster or App Service provisioned.
  • Secrets management (for API keys, connection strings).

Inputs

  • Source code, Dockerfiles, deployment manifests (k8s YAML).

Outputs

  • CI/CD workflow YAML files.
  • Automatic deployments to dev/staging on PR merge.
  • Manual approval gate for production.

Detailed Backend Workflow

  1. Build stage – restore, build, test (unit + integration) for API.
  2. Build and push Docker image for API.
  3. Build Angular app – ng build --prod, serve via Nginx (or push to static web app).
  4. Run E2E tests (Cypress) against the built image.
  5. Deploy to staging – update Kubernetes manifests (or App Service) with new image tags.
  6. Production deployment – trigger on push to main with manual approval.
  7. Notify on success/failure via Slack/Teams.
  8. Commit pipeline YAML to repository.

Data Flow Diagram

Push → Build & Test → Container Registry → Deploy (staging/prod)

Deliverables

  • CI/CD workflow files.
  • Deployment manifests (k8s or App Service).

Validation Checklist

  • ☐ Pipeline runs successfully on PR.
  • ☐ Deployment to staging occurs automatically.
  • ☐ Production deployment requires approval.
  • ☐ Secrets are injected securely.

Testing Strategy

Integration: Test the pipeline by pushing a change and verifying deployment.

Common Pitfalls

Pitfall: Not using deployment slots – can cause downtime. Use blue‑green or rolling updates.

Definition of Done

  • Pipeline fully operational.

Learning Resources

Must Know: GitHub Actions or Azure DevOps YAML, Docker, k8s manifests.

DEV-003Database Migration Pipeline P0 Medium 4h ▾

Task Information

  • Task ID: DEV-003
  • Task Name: Database Migration Pipeline
  • Owner: M5 (DevOps) with M1
  • Dependencies: DEV-002 (CI/CD), DB-010 (Migrations)
  • Complexity: Medium
  • Estimated Effort: 4 hours

Objective

Automatically run EF Core database migrations as part of the deployment pipeline to ensure the database schema is always up‑to‑date with the application version.

Business Purpose

Prevent schema mismatch errors and enable zero‑downtime migrations (if using safe strategies).

Technical Purpose

Integrate dotnet ef database update into the CI/CD pipeline, executed after the API is deployed (or before). Use idempotent migrations.

Prerequisites

  • EF Core tools installed in pipeline.
  • Connection string to the target database (from secrets).

Inputs

  • Migration files.
  • Target environment (staging/prod).

Outputs

  • Migrations applied.

Detailed Backend Workflow

  1. Add a pipeline step after deployment (or before) to run dotnet ef database update --project src/Jalsa.Infrastructure --startup-project src/Jalsa.API.
  2. Use environment‑specific connection strings from secrets.
  3. Ensure idempotency – EF Core migrations are already idempotent.
  4. Rollback strategy – if migration fails, halt deployment and alert.
  5. Test by deploying to staging and verifying migrations apply.

Data Flow Diagram

Deploy API → Run Migration → Database Updated

Deliverables

  • Pipeline YAML updated with migration step.

Validation Checklist

  • ☐ Migrations run successfully in staging.
  • ☐ No errors.
  • ☐ Schema matches expected.

Common Pitfalls

Pitfall: Running migrations during deployment can cause downtime if they are not backward‑compatible. Use safe migration patterns (e.g., adding columns with defaults).

Definition of Done

  • Migration step integrated and working.

Learning Resources

Must Know: EF Core migrations, pipeline scripting.

DEV-004API Deployment (Container) P0 Medium 6h ▾

Task Information

  • Task ID: DEV-004
  • Task Name: API Deployment (Container)
  • Owner: M5 (DevOps)
  • Dependencies: DEV-002 (CI/CD), Dockerfile for API
  • Complexity: Medium
  • Estimated Effort: 6 hours

Objective

Containerise the ASP.NET Core API and deploy it to the target environment (Kubernetes or App Service) via the CI/CD pipeline.

Business Purpose

Ensure the API runs consistently across environments, is scalable, and can be updated with zero downtime.

Technical Purpose

Build a Docker image for the API, push to container registry, and use orchestration (Kubernetes) or App Service to deploy and manage the container.

Prerequisites

  • Dockerfile (from Phase 0).
  • Container registry (ACR or Docker Hub).
  • Deployment target (k8s cluster or App Service).

Inputs

  • API source code, Dockerfile.

Outputs

  • API running in staging/production.

Detailed Backend Workflow

  1. Build Docker image with appropriate tags (e.g., jalsa-api:latest, commit hash).
  2. Push to registry.
  3. Deploy – if using Kubernetes, apply deployment YAML with new image tag; if App Service, update container settings.
  4. Health check – ensure the container is healthy (use /health endpoint).
  5. Rollback – if failed, revert to previous version.

Data Flow Diagram

Build Image → Push → Deploy → Healthy

Deliverables

  • Deployment configuration (k8s manifests or App Service config).

Validation Checklist

  • ☐ API pod is running.
  • ☐ Health check passes.
  • ☐ API responds to requests.

Common Pitfalls

Pitfall: Not exposing correct ports – ensure Dockerfile exposes the correct port (e.g., 8080).

Definition of Done

  • API container running in staging.

Learning Resources

Must Know: Docker, Kubernetes/App Service deployment.

DEV-005Frontend Deployment P0 Medium 5h ▾

Task Information

  • Task ID: DEV-005
  • Task Name: Frontend Deployment
  • Owner: M3 (Frontend) with M5
  • Dependencies: DEV-002 (CI/CD), Angular build
  • Complexity: Medium
  • Estimated Effort: 5 hours

Objective

Build the Angular application and deploy it to a static web hosting service (Azure Static Web Apps, AWS S3, or Nginx container) as part of the CI/CD pipeline.

Business Purpose

Deliver the frontend to users with high availability and low latency.

Technical Purpose

Automate the build and deployment of the Angular app, ensuring environment‑specific configuration (API base URL) is injected.

Prerequisites

  • Angular project.
  • Static hosting service.
  • Environment files (environment.prod.ts).

Inputs

  • Source code.

Outputs

  • Deployed frontend application.

Detailed Frontend Workflow

  1. Build for production – ng build --configuration=production.
  2. Copy build output – if using Azure Static Web Apps, the pipeline can publish directly; if using S3, sync the dist folder.
  3. Alternatively, containerise – build an Nginx image serving the Angular app and deploy as a container (like the API).
  4. Configure environment variables – use placeholders replaced at runtime.
  5. Test – verify the deployed site works.

Data Flow Diagram

Build → Static Host → Live

Deliverables

  • Deployment configuration for frontend.

Validation Checklist

  • ☐ Frontend is accessible via URL.
  • ☐ API calls go to correct backend.
  • ☐ Routing works.

Common Pitfalls

Pitfall: Hard‑coded API URLs – use environment variables.

Definition of Done

  • Frontend deployed in staging.

Learning Resources

Must Know: Angular build, static hosting.

DEV-006Monitoring & Alerting P1 Medium 8h ▾

Task Information

  • Task ID: DEV-006
  • Task Name: Monitoring & Alerting
  • Owner: M5 (DevOps)
  • Dependencies: DEV-004, DEV-005 (deployed services)
  • Complexity: Medium
  • Estimated Effort: 8 hours

Objective

Set up monitoring and alerting for the production system using Application Insights (Azure) or Prometheus + Grafana. Monitor API health, performance, errors, and infrastructure metrics.

Business Purpose

Detect issues early, ensure high availability, and provide visibility into system performance.

Technical Purpose

Collect metrics (request rates, response times, error rates, CPU/memory) and set up alerts for thresholds (e.g., 500 errors > 5% for 5 minutes).

Prerequisites

  • Application Insights or Prometheus/Grafana stack.
  • Logging infrastructure (Serilog sinks).

Inputs

  • Deployed services.

Outputs

  • Monitoring dashboards.
  • Alert rules.
  • Notification channels (email, Slack).

Detailed Backend Workflow

  1. Integrate Application Insights – add Application Insights SDK to API, configure instrumentation key.
  2. Enable request logging – Serilog to App Insights.
  3. Set up custom metrics – e.g., AI service call duration, database query time.
  4. Create dashboards – for key metrics.
  5. Define alerts – e.g., availability < 99.9%, error rate spike, high latency.
  6. Set up action groups – email, SMS, Slack.
  7. Test – trigger a test alert.

Data Flow Diagram

API → Telemetry → App Insights → Alerts → Team

Deliverables

  • Dashboards.
  • Alert rules.
  • Documentation.

Validation Checklist

  • ☐ Metrics are being collected.
  • ☐ Alerts trigger correctly.
  • ☐ Dashboards display key information.

Common Pitfalls

Pitfall: Not setting up alerts – monitoring is useless without alerts. Define meaningful thresholds.

Definition of Done

  • Monitoring and alerting operational.

Learning Resources

Must Know: Application Insights, Prometheus, Grafana.

DEV-008Secrets Management P0 Medium 4h ▾

Task Information

  • Task ID: DEV-008
  • Task Name: Secrets Management
  • Owner: M5 (DevOps) with M1
  • Dependencies: DEV-002 (CI/CD), cloud infrastructure
  • Complexity: Medium
  • Estimated Effort: 4 hours

Objective

Store and manage all secrets (JWT secret, OpenAI API keys, database connection strings, SendGrid keys) securely using Azure Key Vault or AWS Secrets Manager, and inject them into the application at runtime.

Business Purpose

Prevent secrets from being exposed in source code or configuration files, ensuring compliance and security.

Technical Purpose

Centralize secret management, enable rotation, and provide access control.

Prerequisites

  • Key Vault or Secrets Manager provisioned (via IaC).
  • Application code supports reading from environment variables or Azure App Configuration.

Inputs

  • List of secrets (keys).

Outputs

  • Secrets stored in Key Vault.
  • Application configuration to read secrets.
  • Pipeline integration to inject secrets.

Detailed Backend Workflow

  1. Create Key Vault (via IaC).
  2. Add secrets – JwtSecret, OpenAiApiKey, SendGridApiKey, DatabaseConnectionString, etc.
  3. Configure API to use IConfiguration with Key Vault provider (Azure.Extensions.AspNetCore.Configuration.Secrets).
  4. In CI/CD, set environment variables to point to Key Vault or use managed identity.
  5. Test – run application locally with Key Vault access.

Data Flow Diagram

Application → Key Vault → Secrets

Deliverables

  • Key Vault configured.
  • Application integration.
  • Pipeline integration.

Validation Checklist

  • ☐ Secrets are not in source code.
  • ☐ Application loads secrets successfully.
  • ☐ Access is restricted.

Common Pitfalls

Pitfall: Not granting pipeline managed identity access to Key Vault – leads to deployment failure.

Definition of Done

  • Secrets management implemented.

Learning Resources

Must Know: Azure Key Vault, Managed Identity.

DEV-009Production Environment Setup P0 Medium 6h ▾

Task Information

  • Task ID: DEV-009
  • Task Name: Production Environment Setup
  • Owner: M5 (DevOps) with M1, M3
  • Dependencies: DEV-001 (IaC), DEV-004, DEV-005, DEV-008
  • Complexity: Medium
  • Estimated Effort: 6 hours

Objective

Prepare the production environment: configure DNS, SSL certificates, load balancer, WAF (Web Application Firewall), and ensure all security best practices are applied.

Business Purpose

Make the application available to end users with a secure, branded domain, and protect against common attacks.

Technical Purpose

Set up production‑grade networking, HTTPS, and security policies.

Prerequisites

  • Domain name.
  • SSL certificate (e.g., from Let's Encrypt or cloud provider).
  • Load balancer/Application Gateway.

Inputs

  • DNS records, certificate.

Outputs

  • Production URL (e.g., app.jalsa.com).
  • SSL termination.
  • WAF rules configured.

Detailed Backend Workflow

  1. Provision DNS zone – point domain to load balancer IP.
  2. Obtain SSL certificate – use cloud provider's certificate service or Let's Encrypt.
  3. Configure load balancer – set up HTTPS listener, forward to backend pool (API and frontend).
  4. Enable WAF – enable OWASP rules.
  5. Set up health probes – for load balancer.
  6. Test – browse to the domain and verify HTTPS.

Data Flow Diagram

User → DNS → Load Balancer → WAF → Application

Deliverables

  • Production environment live.
  • SSL certificate.
  • WAF rules.

Validation Checklist

  • ☐ Domain resolves correctly.
  • ☐ HTTPS works.
  • ☐ WAF blocks malicious requests.
  • ☐ Health probes pass.

Common Pitfalls

Pitfall: DNS propagation delay – allow time for changes to take effect.

Definition of Done

  • Production environment configured and accessible.

Learning Resources

Must Know: DNS, SSL, load balancing, WAF.

DEV-010Disaster Recovery & Backup P1 Medium 5h ▾

Task Information

  • Task ID: DEV-010
  • Task Name: Disaster Recovery & Backup
  • Owner: M5 (DevOps)
  • Dependencies: DEV-001 (IaC), DEV-009 (Production)
  • Complexity: Medium
  • Estimated Effort: 5 hours

Objective

Implement backup and disaster recovery strategies for the database, blob storage, and configuration, ensuring data can be restored in case of failure.

Business Purpose

Protect patient data and ensure business continuity in case of data loss or service outage.

Technical Purpose

Configure automated database backups (point‑in‑time restore), geo‑redundant storage, and backup of configuration (IaC state). Define RPO and RTO.

Prerequisites

  • Database backup settings (Azure PostgreSQL automated backups).
  • Blob storage replication.
  • Backup of Terraform state.

Inputs

  • Backup policies.

Outputs

  • Backup schedules.
  • Restore procedures.
  • Test restoration plan.

Detailed Backend Workflow

  1. Configure database backup – enable automated backups, set retention (e.g., 7 days).
  2. Configure blob storage – enable versioning and soft delete, set replication (geo‑redundant).
  3. Backup Terraform state – store state in a separate storage account.
  4. Document restore process – step‑by‑step for database, blobs, and application.
  5. Test restoration – restore database to a staging environment to verify.

Data Flow Diagram

Database → Backup → Storage → Restore

Deliverables

  • Backup configuration.
  • Restore documentation.
  • Successful restore test.

Validation Checklist

  • ☐ Backups are running.
  • ☐ Restore procedure tested.
  • ☐ RPO defined and achievable.

Common Pitfalls

Pitfall: Not testing restores – backups are useless if not verified. Schedule regular restore drills.

Definition of Done

  • Backup and DR plan implemented and tested.

Learning Resources

Must Know: Database backup, storage replication, RPO/RTO.
