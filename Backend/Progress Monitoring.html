📈 Progress Monitoring – Phase 8

Purpose: Provide therapists with dashboards and analytics to monitor patient progress – assessment trends, exercise completion rates, session statistics, and overall patient summaries. This phase builds the backend aggregation logic and frontend visualisation.

📋 Tasks: 7 (4 backend + 3 frontend) ⏱️ Total Effort: ~32 hours 👤 Owners: M2 (Backend), M3 (Frontend Lead) 🔗 Dependencies: Phase 3 (Patient), Phase 4 (Session), Phase 5 (Exercise), Phase 6 (AI)

BE-PRO-002Progress Service P1 High 8h ▾

Task Information

  • Task ID: BE-PRO-002
  • Task Name: Progress Service
  • Owner: M2 (Backend)
  • Dependencies: BE-PRO-001 (DTOs), Patient, Session, Exercise, Assessment repositories, AI summary service
  • Complexity: High
  • Estimated Effort: 8 hours

Objective

Implement the progress service that aggregates data from multiple sources to build the therapist dashboard.

Business Purpose

Provide therapists with a holistic view of patient progress to inform clinical decisions.

Technical Purpose

Query and aggregate data from Patient, Session, Exercise, Assessment, and AI summarization services. Use efficient LINQ queries to avoid performance issues.

Prerequisites

  • LINQ aggregation: GroupBy, Count, Average, Sum.
  • EF Core performance: Use AsNoTracking, projection.
  • Repository pattern: Access to data.

Inputs

  • PatientRepository, SessionRepository, ExerciseRepository, AssessmentRepository, ISummarizationService.

Outputs

  • IProgressService interface.
  • ProgressService implementation.

Detailed Backend Workflow

  1. Create IProgressService with method GetDashboardAsync().
  2. Implement GetDashboardAsync:
    • Get total patient count, active (not archived), archived.
    • Get total sessions, sessions in last 30 days.
    • Get exercise completion rate (Complete / total assigned).
    • Get average assessment score across all assessments (group by type).
    • Get recent crisis alerts.
    • Build assessment trend: group assessments by month, average score.
    • Build exercise completion pie: Complete, Partial, Skipped.
    • Build session frequency: sessions per week over last 8 weeks.
  3. Use AsNoTracking() for read‑only queries.
  4. Return aggregated DashboardSummaryDto.

Data Flow

Controller → IProgressService → Repositories → Database → Aggregation → DTO

Deliverables

  • Service interface and implementation.
  • DI registration.

Validation Checklist

  • Counts and percentages are correct.
  • Date ranges work (e.g., last 30 days).
  • Empty data handled gracefully.
  • Performance is acceptable (use AsNoTracking).

Testing Strategy

Unit Tests: Mock repositories, test aggregation logic.

Integration Tests: Use seeded database to verify queries.

Common Pitfalls

Pitfall: Querying large datasets without filtering – always use date ranges and pagination.
Pitfall: Not using AsNoTracking() – can cause memory bloat.

Definition of Done

  • Service implemented and tested.

Learning Resources

Must Know: LINQ aggregation, EF Core performance.

FE-PRO-003Dashboard Component P1 Medium 8h ▾

Task Information

  • Task ID: FE-PRO-003
  • Task Name: Dashboard Component
  • Owner: M3 (Frontend Lead)
  • Dependencies: FE-PRO-002
  • Complexity: Medium
  • Estimated Effort: 8 hours

Objective

Create the dashboard component with charts for assessment trends, exercise completion, and session statistics.

Business Purpose

Visualise patient progress data for therapists.

Technical Purpose

Use Chart.js or ngx‑charts to render line charts, bar charts, and pie charts from dashboard data.

Prerequisites

  • Charting library (Chart.js / ngx‑charts).
  • DashboardService.

Inputs

  • Dashboard data from service.

Outputs

  • DashboardComponent with template and charts.

Detailed Frontend Workflow

  1. Generate component: ng g c dashboard/dashboard.
  2. On init, load dashboard data.
  3. Show loading spinner.
  4. Display summary cards: Total Patients, Active Patients, Sessions This Month, Exercise Completion Rate.
  5. Render assessment trend line chart (scores over time).
  6. Render exercise completion pie chart (Complete/Partial/Skipped).
  7. Render session frequency bar chart (sessions per week).
  8. Display recent crisis alerts list.

Data Flow

Component → DashboardService → API → Data → Charts

Deliverables

  • Dashboard component with charts.

Validation Checklist

  • Data loads.
  • Cards display correct numbers.
  • Charts render with data.
  • Loading and error states handled.
  • Responsive design.

Testing Strategy

Unit Tests: Test component with mock data.

E2E Tests: Verify dashboard renders.

Common Pitfalls

Pitfall: Not handling empty data – charts should show "no data" message.
Pitfall: Chart library not properly configured – ensure correct imports and options.

Definition of Done

  • Dashboard component functional and charts render.

Learning Resources

Must Know: Chart.js / ngx‑charts integration.

Nice To Know: Angular animations for charts.
