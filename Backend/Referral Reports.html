📄 Referral Reports – Phase 7

Purpose: Enable therapists to generate AI‑drafted referral reports, review, edit, approve/reject, and export as PDF. This phase includes backend report generation, approval workflow, PDF export, and frontend report management.

📋 Tasks: 13 (8 backend + 5 frontend) ⏱️ Total Effort: ~58 hours 👤 Owners: M1 (Backend Lead), M4 (Frontend) 🔗 Dependencies: Phase 3 (Patient), Phase 4 (Session), Phase 6 (AI Services)

  • Task ID: BE-REP-001
  • Task Name: Report DTOs
  • Owner: M1 (Backend Lead)
  • Dependencies: DB-006 (ReferralReport entity)
  • Complexity: Low
  • Estimated Effort: 3 hours

Objective

Create DTOs for referral report operations: ReportCreateDto, ReportUpdateDto, ReportViewDto, ApprovalDto.

Business Purpose

Define data contracts for generating, reviewing, editing, and approving reports.

Technical Purpose

Provide type‑safe communication, validation, and separation from domain entities.

Prerequisites

  • DTO pattern: Understand role in Clean Architecture.

Inputs

  • ReferralReport entity from Phase 1.

  • ApprovalDto – ReportId, Status (Approved/Rejected), Comments (optional).

Detailed Backend Workflow

  1. Create DTO classes in Jalsa.Application/DTOs/Report/.
  2. Add validation: Status enum validation, Content length constraints.

Deliverables

Validation Checklist

  • String lengths constrained.
  • Enum validation.

Testing Strategy

BE-REP-003Report Service P0 High 6h ▾

Task Information

  • Task ID: BE-REP-003
  • Task Name: Report Service
  • Owner: M1 (Backend Lead)
  • Dependencies: BE-REP-001 (DTOs), BE-REP-002 (Repository), BE-AI-006 (Report Draft), AutoMapper
  • Complexity: High
  • Estimated Effort: 6 hours

Objective

Implement the report service layer with methods for generating drafts, retrieving, editing, approving, rejecting, and viewing history.

Business Purpose

Encapsulate business logic for the complete report workflow.

Technical Purpose

Coordinate AI generation, repository operations, and audit logging (ReportHistory).

Prerequisites

  • AutoMapper, FluentValidation.
  • UnitOfWork.
  • AI report generation service.

Inputs

  • IReportRepository, IReportGenerationService, IMapper, IUnitOfWork.

Outputs

  • IReportService interface.
  • ReportService implementation.

Detailed Backend Workflow

  1. Create IReportService with methods: GenerateDraftAsync(patientId), GetByIdAsync(id), GetByPatientAsync(patientId), UpdateReportAsync(updateDto), ApproveReportAsync(approvalDto), RejectReportAsync(approvalDto), GetHistoryAsync(reportId).
  2. Implement GenerateDraftAsync: call AI service to get draft, create Report entity with status Draft, save.
  3. Update: allow editing draft or final content, save history snapshot.
  4. Approve: set status Approved, set FinalContent = DraftContent (or keep separate).
  5. Reject: set status Rejected, add comments.
  6. History: each update/approval creates ReportHistory record.

Data Flow

Controller → IReportService → Repository + AI → Database

Deliverables

  • Service interface and implementation.
  • AutoMapper profiles.

Validation Checklist

  • Generate creates draft.
  • Update saves history.
  • Approve and reject set correct status.

Testing Strategy

Unit Tests: Mock repository and AI service, test each method.

Definition of Done

  • Service implemented and unit tested.

BE-REP-005PDF Export P1 Medium 6h ▾

Task Information

  • Task ID: BE-REP-005
  • Task Name: PDF Export
  • Owner: M1 (Backend Lead)
  • Dependencies: BE-REP-003 (ReportService), QuestPDF or iTextSharp
  • Complexity: Medium
  • Estimated Effort: 6 hours

Objective

Generate a professional PDF document from a finalized referral report, store it in blob storage, and return the PDF to the client.

Business Purpose

Provide a shareable, printable version of the report for external providers.

Technical Purpose

Use QuestPDF or iTextSharp to create a well‑formatted PDF with header, sections, and styling. Upload the PDF to blob and update the Report entity with the URL.

Prerequisites

  • QuestPDF / iTextSharp library.
  • Blob storage service.

Inputs

  • Report ID (final content).
  • Patient and therapist information.

Outputs

  • PDF file (stream).
  • PdfUrl updated in Report entity.

Detailed Backend Workflow

  1. Create IPdfGenerator service.
  2. Implement GeneratePdfAsync(reportId):
    • Retrieve report, patient, therapist.
    • Build PDF document with header (clinic name, title, date), sections (Patient Info, History, Assessments, Sessions, Recommendations).
    • Use QuestPDF fluent API to define layout.
    • Generate byte array.
    • Upload to blob storage, get URL.
    • Update Report.PdfUrl.
    • Return byte array (or stream) for download.
  3. Integrate with ReportController export endpoint.

Data Flow

Report ID → PDF Generator → Blob Storage → Client

Deliverables

  • PDF generator service.
  • Integration with report export.

Validation Checklist

  • PDF is generated with correct content.
  • File is stored and URL is saved.
  • Download works.

Common Pitfalls

Pitfall: Hard‑coded fonts or paths – use embedded fonts and relative paths.

Testing Strategy

Unit Tests: Test PDF generation with mock data.

Integration Tests: Test upload and retrieval.

Definition of Done

  • PDF export works.
