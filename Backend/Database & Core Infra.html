📦 Phase 1 – Database & Core Infrastructure

Purpose: Design and implement the database schema using EF Core Code First, configure repositories, set up the vector store, and establish the data access layer. This phase lays the foundation for all data persistence and retrieval.

DB-001 Domain Entities – Core P0 Medium 4h ▾

Task Information

  • Task ID: DB-001
  • Task Name: Domain Entities – Core
  • Owner: M1 (Backend Lead)
  • Dependencies: Phase 0 (Foundation) – solution structure exists
  • Complexity: Medium
  • Estimated Effort: 4 hours

Objective

Create the core domain entity classes: User, Role, and Permission (if used). These form the authentication and authorisation base.

Business Purpose

All users (therapists and patients) need to be stored and authenticated. The User entity holds credentials and profile information. Role defines the type of user (Therapist, Patient).

Technical Purpose

Establish the foundation for JWT authentication and role‑based authorization. These entities are referenced by many other entities (e.g., PatientProfile references User, SessionNote references User as therapist).

Prerequisites

Backend

  • Entity Framework Core: Understanding of entities, properties, and navigation properties.
  • Clean Architecture: Domain layer should contain only plain C# classes with no external dependencies.
  • Why needed: To correctly model the entities.
  • Usage: These entities will be used in all layers.

Frontend

  • None – this is backend data model.

Database

  • Database design: Understanding of primary keys, foreign keys, and relationships.
  • Why needed: To define relationships (e.g., User has one Role).
  • Usage: These entities will be mapped to tables.

Inputs

  • Phase 0 solution with Jalsa.Domain project.
  • Requirement: Users have email, password hash, first name, last name, role.

Outputs

  • User entity class.
  • Role entity class.
  • Permission entity (optional, if needed).
  • Base entity with audit fields (CreatedAt, UpdatedAt, IsDeleted) – to be inherited.

Detailed Backend Workflow

  1. Create Base Entity – In Jalsa.Domain/Common/BaseEntity.cs, define properties: Id (Guid), CreatedAt (DateTime), UpdatedAt (DateTime), IsDeleted (bool). Use Guid.NewGuid() for Id generation.
    Why: Provides consistent audit fields and soft‑delete support.
  2. Create Role entity – In Jalsa.Domain/Entities/Role.cs, define Id (Guid), Name (string, required, max 50), Description (string, optional). Add navigation property ICollection<User> Users.
  3. Create User entity – In Jalsa.Domain/Entities/User.cs, inherit from BaseEntity. Add properties: Email (string, required, unique), PasswordHash (string, required), FirstName (string, required, max 100), LastName (string, required, max 100), RoleId (Guid, foreign key). Add navigation property Role.
  4. Optional: Permission entity – If fine‑grained permissions are needed, add Permission and a many‑to‑many relationship with Role. For MVP, we'll keep it simple with roles.
  5. Ensure all properties are properly typed – Use string for text, DateTime for dates, bool for flags.
  6. Add XML comments – Document each property for clarity.

Detailed Database Workflow

This task creates the domain classes; database mappings will be configured in DB-009. However, we anticipate the following:

  • Tables: Users, Roles.
  • Relationships: User has one Role (foreign key RoleId).
  • Constraints: Email unique, PasswordHash not null.
  • Indexes: On Email for fast login.

Detailed Frontend Workflow

None – this is backend only.

API Design

No endpoints yet – these entities are for internal storage.

Data Flow Diagram

Application → User/Role Entities → DbContext → Database

Deliverables

  • Jalsa.Domain/Common/BaseEntity.cs
  • Jalsa.Domain/Entities/User.cs
  • Jalsa.Domain/Entities/Role.cs
  • Optional Permission.cs if needed.

Validation Checklist

  • All entities inherit from BaseEntity (or have Id).
  • Navigation properties are correctly typed (Role for User, ICollection<User> for Role).
  • Properties have appropriate data types and constraints (e.g., string with Required).
  • No database‑specific attributes are used – keep domain pure.
  • Project builds with no errors.

Testing Strategy

Unit Tests: Not applicable – these are simple POCOs. However, later integration tests will verify mappings.

Integration Tests: Will be added in DB-011 when repositories are used.

Common Pitfalls

Pitfall: Using List<T> for navigation properties – use ICollection<T> to avoid EF Core lazy‑loading issues.
Pitfall: Adding foreign key properties as int instead of Guid – use Guid for consistency.
Tip: Use private set for Id to enforce encapsulation, or use a base class constructor to assign Id.

Definition of Done

  • All core entity classes are created in the Domain project.
  • BaseEntity is implemented.
  • Navigation properties are defined.
  • Solution builds successfully.

Learning Resources

Must Know

  • EF Core entities and navigation properties.
  • Guid usage for primary keys.

Nice To Know

  • Value objects (e.g., Email as a value object).

  2. Clean Architecture domain layer (1h).

DB-009 DbContext Configuration P0 High 6h ▾

Task Information

  • Task ID: DB-009
  • Task Name: DbContext Configuration
  • Owner: M1 (Backend Lead)
  • Dependencies: DB-001 through DB-008 (all entities)
  • Complexity: High
  • Estimated Effort: 6 hours

Objective

Create the EF Core DbContext class and configure all entity mappings using Fluent API.

Business Purpose

This is the bridge between domain entities and the database. Correct configuration ensures data integrity and performance.

Technical Purpose

Define table names, relationships, constraints, indexes, and soft‑delete filters. Enable migrations.

Prerequisites

  • EF Core: Fluent API, OnModelCreating, DbSet properties.
  • Database design: Primary keys, foreign keys, indexes.

Inputs

  • All entity classes from DB-001..008.

Outputs

  • JalsaDbContext class in Infrastructure/Data.
  • Service registration for DbContext (AddDbContext).

Detailed Backend Workflow

  1. Create JalsaDbContext – Inherit from DbContext.
  2. Add DbSet<T> for each entity.
  3. Override OnModelCreating – Use modelBuilder.Entity<T>() to configure:
    • Table names (e.g., .ToTable("Users"))
    • Primary keys (.HasKey(x => x.Id))
    • Relationships – .HasOne(x => x.Role).WithMany(r => r.Users).HasForeignKey(x => x.RoleId)
    • Constraints – .Property(x => x.Email).IsRequired().HasMaxLength(256)
    • Indexes – .HasIndex(x => x.Email).IsUnique()
    • Global query filters for soft‑delete – .HasQueryFilter(x => !x.IsDeleted)
  4. Add OnConfiguring – If not using DI, but we'll use DI.
  5. Register DbContext in Startup.cs / Program.cs: services.AddDbContext<JalsaDbContext>(options => options.UseNpgsql(connectionString));
  6. Add migration (later task).

Detailed Database Workflow

  • All tables are defined with proper constraints.
  • Foreign keys are configured with OnDelete behaviour (e.g., Restrict to prevent accidental deletion).
  • Indexes are added for frequently queried columns.

Data Flow

Application → DbContext → Database

Deliverables

  • JalsaDbContext.cs
  • Service registration in DI.

Validation Checklist

  • All DbSet properties exist.
  • Fluent API covers all relationships.
  • Soft‑delete filter applied.
  • Indexes defined.

Common Pitfalls

Pitfall: Forgetting to set OnDelete behaviour – can cause cascade delete loops. Use Restrict for critical relationships.
Pitfall: Not configuring HasQueryFilter for soft‑delete – then deleted records will appear in queries.

Definition of Done

  • DbContext is created and configured.
  • DI registration is complete.
  • Solution builds.

Learning Resources

  • EF Core Fluent API documentation.
  • Entity relationships in EF Core.

Objective

Implement the generic repository pattern and Unit of Work to abstract data access.

Business Purpose

Simplifies data access code, enables mocking for unit tests, and centralises common query logic.

Technical Purpose

Create IGenericRepository<T> and GenericRepository<T> with methods like GetByIdAsync, AddAsync, Update, Delete, GetAllAsync. Also IUnitOfWork to coordinate multiple repositories.

Prerequisites

  • Understanding of generics and dependency injection.
  • EF Core DbContext.

Inputs

Outputs

  • IGenericRepository<T> interface.
  • GenericRepository<T> implementation.
  • IUnitOfWork interface and UnitOfWork implementation.
  • DI registration.

Detailed Backend Workflow

  1. Create IGenericRepository<T> – Define methods: GetByIdAsync, GetAllAsync, FindAsync (with expression), AddAsync, Update, Delete.
  2. Implement GenericRepository<T> – Inject DbContext, use DbSet<T>. Implement methods.
  3. Create IUnitOfWork – Expose repositories (e.g., IPatientRepository later), SaveChangesAsync.
  4. Implement UnitOfWork – Wrap all repository instances, use same DbContext.
  5. Register in DI – services.AddScoped<IUnitOfWork, UnitOfWork>().

Data Flow

Service → IUnitOfWork → Repository → DbContext → Database

Deliverables

  • Repository interfaces and implementations.
  • UnitOfWork.
  • DI registration.

Validation Checklist

  • UnitOfWork commits changes correctly.
  • DI resolves dependencies.

Common Pitfalls

Pitfall: Using Dispose incorrectly – ensure DbContext lifetime is scoped.

Definition of Done

  • Unit tests for repository pass.

DB-014 Vector Store Setup P1 High 5h ▾

Task Information

  • Task ID: DB-014
  • Task Name: Vector Store Setup
  • Owner: M5 (DevOps / QA / AI)
  • Dependencies: DB-009 (DbContext) – but can be separate
  • Complexity: High
  • Estimated Effort: 5 hours

Objective

Set up the vector database (pgvector or Azure Cognitive Search) for storing and querying embeddings.

Business Purpose

Enable semantic search and RAG (Retrieval Augmented Generation) for AI features like report generation and chat.

Technical Purpose

Install and configure pgvector extension on PostgreSQL, or configure Azure Cognitive Search index. Create collections for session notes embeddings.

Prerequisites

  • Vector databases: Understanding of embeddings, similarity search.
  • PostgreSQL: Ability to install extensions.

Inputs

  • PostgreSQL instance (from Docker Compose).

Outputs

  • pgvector extension enabled.
  • Table or collection for embeddings (e.g., Embeddings table with vector column).
  • Configuration for vector store client.

Detailed Backend Workflow

  1. Enable pgvector extension – CREATE EXTENSION IF NOT EXISTS vector;.
  2. Create embeddings table – CREATE TABLE Embeddings (Id uuid, ChunkText text, Embedding vector(1536), Metadata jsonb).
  3. Create index for cosine similarity – CREATE INDEX ON Embeddings USING ivfflat (embedding vector_cosine_ops);.
  4. Create a service class – IVectorStore with methods AddEmbedding, SimilaritySearch.
  5. Implement using Npgsql – or use Azure SDK if using Cognitive Search.
  6. Register in DI.

Detailed Database Workflow

  • Table: Embeddings.
  • Vector column: embedding vector(1536) for OpenAI embeddings.
