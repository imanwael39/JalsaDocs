📊 Chat Monitoring & Crisis Alerts – Phase 10

Purpose: Provide therapists with visibility into patient chat activity, including engagement summaries and crisis alerts. This phase builds the backend endpoints for chat monitoring and the frontend for therapists to view and act on alerts. It also ensures crisis notifications are delivered via the notification service.

📋 Tasks: 4 (all backend) ⏱️ Total Effort: ~12 hours 👤 Owners: M5 (AI/DevOps) + M1 (Backend) 🔗 Dependencies: Phase 9 (Patient Portal – Chat), Phase 6 (AI Services – Crisis detection), Phase 2 (Authentication)
BE-CHM-001Chat Engagement Summary P1 Medium 5h ▾

Task Information

  • Task ID: BE-CHM-001
  • Task Name: Chat Engagement Summary
  • Owner: M5 (AI/DevOps) with M1
  • Dependencies: DB-005 (ChatMessage, ChatSession, CrisisAlert), Patient repository
  • Complexity: Medium
  • Estimated Effort: 5 hours

Objective

Build a service that aggregates chat metrics per patient – total messages, average messages per day, sentiment distribution, number of crisis flags, and last message timestamp – to provide a summary for therapists.

Business Purpose

Give therapists an overview of patient chat activity to identify patients who are engaging well or may need intervention.

Technical Purpose

Query the ChatMessage and CrisisAlert tables, group by patient, compute aggregates, and return a structured DTO for display in the frontend monitoring dashboard.

Prerequisites

Backend

  • LINQ to Entities: GroupBy, Count, Average, OrderBy.
  • Repository pattern: Access to ChatMessage, CrisisAlert, Patient repositories.
  • EF Core performance: Use AsNoTracking for read‑only queries.

Frontend

  • None – this is backend data provision.

Database

  • Tables: ChatMessages, CrisisAlerts, Patients.
  • Indexes on ChatMessage.SessionId, Timestamp, Sender; CrisisAlert.PatientId, ResolvedAt.

AI

  • Not directly used – sentiment analysis may be pre‑computed or can be added later.

Inputs

  • Optional query parameters: date range, patient filter, minimum message count.
  • Patient repository to get patient names.

Outputs

  • ChatEngagementSummaryDto for each patient: PatientId, PatientName, TotalMessages, AverageMessagesPerDay, SentimentDistribution (Positive/Neutral/Negative), CrisisFlagsCount, LastMessageTimestamp, IsActive.

Detailed Backend Workflow

  1. Create DTO ChatEngagementSummaryDto in Jalsa.Application/DTOs/Chat/.
  2. Create IChatMonitoringService interface with method GetEngagementSummariesAsync(DateTime? from, DateTime? to).
  3. Implement ChatMonitoringService:
    • Get all patients (or filter by active).
    • For each patient, query ChatMessages, count, group by date to get average, etc.
    • Use a single query with grouping to avoid N+1: join ChatMessages and CrisisAlerts grouped by PatientId.
    • Compute sentiment distribution (if sentiment column exists, otherwise skip or compute later).
    • Return list of DTOs.
  4. Register service in DI: services.AddScoped<IChatMonitoringService, ChatMonitoringService>();

Detailed Database Workflow

  • Use GroupBy on PatientId, join with Patient table for names.
  • Indexes: ChatMessage.PatientId (if denormalized) or use Session to get Patient; preferably add PatientId to ChatMessage for faster aggregation.
  • If no PatientId on ChatMessage, join through ChatSession.
  • Consider adding a computed column for average if performance is critical.

API Design

RouteMethodRequestResponseAuth
/api/chat-monitoring/engagementGETQuery: from, to, patientId (optional)List<ChatEngagementSummaryDto>Therapist

Validation Rules: Date range validation (from <= to), patientId must be valid Guid if provided.

Authorization Rules: Requires Therapist role.

Data Flow Diagram

Therapist → GET /engagement → ChatMonitoringService → Repository → Database → Aggregation → Response

Deliverables

  • ChatEngagementSummaryDto.
  • IChatMonitoringService interface and implementation.
  • Endpoint in ChatMonitoringController.
  • Unit tests for service.

Validation Checklist

  • ☐ Aggregates correctly calculate total messages.
  • ☐ Average messages per day is correct.
  • ☐ Crisis flags count matches CrisisAlert entries.
  • ☐ Empty data handled gracefully.

Testing Strategy

Unit Tests: Mock repositories, test aggregation logic with sample data.

Integration Tests: Use seeded database to verify query results.

Common Pitfalls

Pitfall: N+1 queries – use eager loading and grouping to reduce database round trips.
Pitfall: Not handling time zones – use UTC consistently.

Definition of Done

  • Service and endpoint implemented.
  • Unit tests pass.

Learning Resources

Must Know: LINQ GroupBy, aggregation, EF Core performance.

Nice To Know: Window functions in SQL for more efficient aggregation.

BE-CHM-002Chat Monitoring Controller P1 Medium 4h ▾

Task Information

  • Task ID: BE-CHM-002
  • Task Name: Chat Monitoring Controller
  • Owner: M1 (Backend)
  • Dependencies: BE-CHM-001 (ChatMonitoringService), BE-AI-010 (Crisis escalation service)
  • Complexity: Medium
  • Estimated Effort: 4 hours

Objective

Create the API endpoints for therapists to view chat engagement summaries and crisis alerts. Also provide an endpoint to mark a crisis alert as resolved.

Business Purpose

Give therapists a dashboard to monitor patient chat activity and respond to crises.

Technical Purpose

Expose chat monitoring data via REST endpoints, with appropriate authorization (Therapist only).

Prerequisites

  • ChatMonitoringService, CrisisAlert repository.
  • Authorization (Therapist role).

Inputs

  • Engagement summary endpoint: query parameters as defined.
  • Crisis alerts endpoint: optional patientId, resolved/unresolved filter.
  • Resolve endpoint: crisis alert ID.

Outputs

  • Engagement summary list.
  • Crisis alert list with details (patient, message snippet, severity, triggeredAt).
  • Success response on resolving.

Detailed Backend Workflow

  1. Create ChatMonitoringController with route [Route("api/chat-monitoring")].
  2. Inject IChatMonitoringService and ICrisisAlertRepository.
  3. Implement GET /engagement – call service, return list.
  4. Implement GET /crisis-alerts – query CrisisAlerts, include patient and chat message, return DTOs. Filter by resolved/unresolved.
  5. Implement PUT /crisis-alerts/{id}/resolve – set ResolvedAt to now, update via repository, return 200.
  6. Apply [Authorize(Roles = "Therapist")] to all endpoints.

API Design

RouteMethodRequestResponseAuth
/api/chat-monitoring/engagementGETQuery: from, to, patientIdList<ChatEngagementSummaryDto>Therapist
/api/chat-monitoring/crisis-alertsGETQuery: resolved (bool), patientIdList<CrisisAlertViewDto>Therapist
/api/chat-monitoring/crisis-alerts/{id}/resolvePUTNone200 OKTherapist

CrisisAlertViewDto: Id, PatientId, PatientName, MessageSnippet, Severity, TriggeredAt, ResolvedAt, ChatMessageId.

Data Flow Diagram

Therapist → Controller → Service/Repository → Database → Response

Deliverables

  • ChatMonitoringController with three endpoints.
  • DTOs for crisis alerts.
  • Unit tests for controller actions.

Validation Checklist

  • ☐ Engagement endpoint returns data.
  • ☐ Crisis alerts filter by resolved state.
  • ☐ Resolve endpoint updates alert.
  • ☐ Authorization enforced.

Testing Strategy

Unit Tests: Mock services and test controller actions.

Integration Tests: Test endpoints with test database.

Definition of Done

  • Controller endpoints implemented and tested.
BE-CHM-003Crisis Alert – Receive P1 Medium 3h ▾

Task Information

  • Task ID: BE-CHM-003
  • Task Name: Crisis Alert – Receive
  • Owner: M5 (AI/DevOps) with M1
  • Dependencies: BE-AI-010 (Crisis escalation), Notification service (EXT-NOT-001), CrisisAlert entity
  • Complexity: Medium
  • Estimated Effort: 3 hours

Objective

Implement the mechanism for therapists to receive crisis alerts (via email, SMS, or push notification) when a crisis is detected (either automatically or manually triggered).

Business Purpose

Ensure therapists are immediately notified of potential patient crises so they can intervene promptly.

Technical Purpose

Integrate the notification service to send alerts to the therapist(s) associated with the patient. The notification should include patient name, severity, and a link to the chat monitoring dashboard.

Prerequisites

  • Notification service (SendGrid/Twilio) already configured.
  • CrisisAlert entity.
  • Patient-therapist relationship (e.g., assigned therapist).

Inputs

  • CrisisAlert ID or PatientId, Severity, Message snippet.

Outputs

  • Notification sent to therapist(s).
  • Log of notification delivery.

Detailed Backend Workflow

  1. Extend INotificationService with method SendCrisisAlertAsync(Guid patientId, string severity, string message).
  2. Implement escalation logic in CrisisEscalationService (or in BE-AI-010). When a crisis is detected:
    • Create CrisisAlert record.
    • Determine the assigned therapist(s) for the patient.
    • Call INotificationService.SendCrisisAlertAsync for each therapist.
    • Log notification.
  3. Ensure idempotency – avoid duplicate notifications for the same crisis.
  4. Handle failures – retry or fallback to alternative channel.

Detailed Database Workflow

  • No new tables – use existing CrisisAlert and Notification logs (if any).

Detailed AI Workflow

This task is about notification delivery, not AI. The AI detection is already done in BE-AI-009/010.

API Design

No new endpoints – this is an internal service.

Data Flow Diagram

Crisis Detected → Create Alert → Find Therapist → Notification Service → Therapist

Deliverables

  • Integration of notification service with crisis escalation.
  • Unit tests for notification sending (mocked).

Validation Checklist

  • ☐ Therapist receives notification.
  • ☐ Notification contains patient name, severity, link.
  • ☐ Retry logic works on failure.

Testing Strategy

Unit Tests: Mock INotificationService and test escalation logic.

Integration Tests: Use a test email/SMS service.

Common Pitfalls

Pitfall: Sending duplicate notifications – track sent status or use idempotency keys.
Pitfall: Notification delay – use async processing (Hangfire) to avoid blocking.

Definition of Done

  • Crisis alerts are delivered to therapists.

Learning Resources

Must Know: Notification service integration, retry policies.
