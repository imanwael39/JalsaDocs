🔑 Patient Portal (Auth + Exercise + Chat) – Phase 9

Purpose: Enable patients to log in, view and update their exercises, and communicate via an AI‑powered support chat. This phase builds the patient‑facing backend for authentication, exercise tracking, and chat with crisis detection.

📋 Tasks: 8 (all backend) ⏱️ Total Effort: ~30 hours 👤 Owners: M2 (Backend) + M5 (AI/DevOps) 🔗 Dependencies: Phase 2 (Authentication), Phase 5 (Exercise), Phase 6 (AI Services), Phase 3 (Patient)
BE-PATPT-001Patient Auth Endpoints P0 Medium 3h ▾

Task Information

  • Task ID: BE-PATPT-001
  • Task Name: Patient Auth Endpoints
  • Owner: M2 (Backend)
  • Dependencies: BE-AUTH-003 (AuthService), BE-AUTH-009 (Patient login already done?) Actually BE-AUTH-009 implemented patient login/ logout, but we need separate endpoints for patient portal. We'll reuse or extend.
  • Complexity: Medium
  • Estimated Effort: 3 hours

Objective

Provide dedicated endpoints for patient login and logout, separate from therapist authentication, to be used by the patient portal frontend. Ensure tokens include the "Patient" role and restrict access to patient-specific resources.

Business Purpose

Patients need a secure way to access their own data (exercises, chat) without therapist privileges. This task enables that.

Technical Purpose

Extend the existing authentication service to handle patient login, return a JWT with the Patient role, and optionally implement a separate logout (client-side token removal).

Prerequisites

Backend

  • JWT Authentication: Already implemented in Phase 2.
  • AuthService: Login, token generation.
  • User entity: Role field.

Frontend

  • None – this is backend.

Database

  • Users table, Roles table.

AI

  • Not applicable.

Inputs

  • LoginRequest DTO (Email, Password).
  • AuthService.

Outputs

  • Controller endpoint: POST /api/patient/auth/login.
  • Returns TokenResponse with Patient role.
  • Optional POST /api/patient/auth/logout (client-side).

Detailed Backend Workflow

  1. Create PatientAuthController in Jalsa.API/Controllers/ with route [Route("api/patient/auth")].
  2. Inject IAuthService.
  3. Implement Login endpoint: call authService.LoginAsync(loginRequest), but ensure the user has role "Patient". If not, return 403 Forbidden.
  4. Implement Logout endpoint (optional – client can clear token). If implemented, simply return 204.
  5. Apply [AllowAnonymous] for login, since it's public.
  6. Add validation using FluentValidation for LoginRequest.

Detailed Database Workflow

No new tables; uses existing Users and Roles.

Detailed Frontend Workflow

Frontend will call these endpoints for patient login; frontend handles storing token and redirecting.

API Design

RouteMethodRequestResponseAuth
/api/patient/auth/loginPOSTLoginRequestTokenResponseNone
/api/patient/auth/logoutPOSTNone204 No ContentAuthorized (Patient)

Validation Rules: Email required, valid email; Password required.

Authorization Rules: Login public; Logout requires Patient role.

Data Flow Diagram

Patient → /login → AuthService → DB (User) → Token → Patient

Deliverables

  • PatientAuthController.
  • Endpoint documentation (Swagger).
  • Unit tests for login (success, wrong password, patient role check).

Validation Checklist

  • ☐ Login returns token with Patient role.
  • ☐ Invalid credentials return 401.
  • ☐ Therapist user trying to login via patient endpoint returns 403.
  • ☐ Logout endpoint (if implemented) requires authentication.

Testing Strategy

Unit Tests

  • Test login with valid patient credentials.
  • Test login with invalid password.
  • Test login with therapist account – expect 403.
  • Test logout – verify 204.

Integration Tests

  • End-to-end login flow with test database.

Common Pitfalls

Pitfall: Not checking the user's role before issuing token – could allow therapist to access patient endpoints. Always verify role.

Definition of Done

  • Endpoint implemented and tested.
  • Swagger updated.
  • Unit tests pass.

Learning Resources

Must Know: JWT authentication, ASP.NET Core controllers, role‑based authorization.

BE-PATPT-002Patient Exercise – View P1 Medium 3h ▾

Task Information

  • Task ID: BE-PATPT-002
  • Task Name: Patient Exercise – View
  • Owner: M2 (Backend)
  • Dependencies: BE-EXE-003 (ExerciseService), BE-AUTH-009 (Patient auth)
  • Complexity: Medium
  • Estimated Effort: 3 hours

Objective

Provide an endpoint for patients to view their assigned exercises, with optional filtering by status (e.g., pending, completed).

Business Purpose

Patients need to see what exercises are assigned to them, their due dates, and their current status.

Technical Purpose

Expose patient‑specific exercise data from the ExerciseService, filtering by the authenticated patient's ID.

Prerequisites

Backend

  • ExerciseService with method to get patient exercises (already implemented in BE-EXE-003).
  • CurrentUserService to extract patient ID from JWT.

Inputs

  • Patient ID from token (via ICurrentUserService).
  • Optional query parameters: status (Pending, Complete, etc.).

Outputs

  • List of PatientExerciseViewDto for the patient.

Detailed Backend Workflow

  1. Create endpoint GET /api/patient/exercises in a new PatientController (or a dedicated PatientPortalController) with route [Route("api/patient")].
  2. Inject IExerciseService, ICurrentUserService.
  3. Get patient ID from ICurrentUserService.GetUserId().
  4. Call exerciseService.GetPatientExercisesAsync(patientId, statusFilter) (if filter is provided).
  5. Return list of DTOs.
  6. Apply [Authorize(Roles = "Patient")].

Detailed Database Workflow

Uses existing PatientExercise table with indexes on PatientId, Status for performance.

API Design

RouteMethodRequestResponseAuth
/api/patient/exercisesGETQuery: status (optional)List<PatientExerciseViewDto>Patient

Response DTO: PatientExerciseViewDto includes ExerciseId, Title, Description, DueDate, Status, Reflection.

Data Flow Diagram

Patient → /api/patient/exercises → Get patient ID → ExerciseService → Repository → Response

Deliverables

  • PatientController with endpoint.
  • Unit tests for service call and authorization.

Validation Checklist

  • ☐ Endpoint returns only exercises for the authenticated patient.
  • ☐ Filter by status works.
  • ☐ 401 if not authenticated.

Testing Strategy

Unit Tests: Mock ICurrentUserService, ExerciseService, test endpoint.

Integration Tests: Test with a real patient and exercises.

Definition of Done

  • Endpoint implemented and tested.
BE-PATPT-003Patient Exercise – Update Status P1 Medium 4h ▾

Task Information

  • Task ID: BE-PATPT-003
  • Task Name: Patient Exercise – Update Status
  • Owner: M2 (Backend)
  • Dependencies: BE-EXE-003 (ExerciseService), BE-PATPT-002
  • Complexity: Medium
  • Estimated Effort: 4 hours

Objective

Allow patients to update the status of an assigned exercise (Complete/Partial/Skipped) and optionally add a reflection note.

Business Purpose

Patients can track their progress and provide feedback on exercises.

Technical Purpose

Call ExerciseService to update the PatientExercise status, ensuring the exercise belongs to the authenticated patient.

Prerequisites

  • ExerciseService with UpdatePatientExerciseStatusAsync method.
  • Ownership validation: ensure the patient owns the exercise.

Inputs

  • PatientExercise ID (from route).
  • CompletionDto (Status, Reflection).
  • Patient ID from token.

Outputs

  • Updated PatientExerciseViewDto.

Detailed Backend Workflow

  1. Create endpoint PUT /api/patient/exercises/{id}/status.
  2. Retrieve patient ID from token.
  3. Fetch the PatientExercise by id to verify ownership – if not owned by patient, return 403.
  4. Call ExerciseService.UpdateStatusAsync with the DTO.
  5. Return updated DTO.

API Design

RouteMethodRequestResponseAuth
/api/patient/exercises/{id}/statusPUTCompletionDto (Status, Reflection)PatientExerciseViewDtoPatient

Validation Rules: Status must be one of Complete, Partial, Skipped; Reflection max length 1000.

Data Flow Diagram

Patient → PUT /status → Verify ownership → ExerciseService → Repository → Response

Deliverables

  • Endpoint in PatientController.
  • Unit tests for ownership and update.

Validation Checklist

  • ☐ Only patient can update their own exercise.
  • ☐ Invalid status returns 400.
  • ☐ Reflection saved.

Testing Strategy

Unit Tests: Test ownership check, update success.

Integration Tests: Test full flow.

Definition of Done

  • Endpoint works.

BE-PATPT-005Support Chat – Send Message P1 High 6h ▾

Task Information

  • Task ID: BE-PATPT-005
  • Task Name: Support Chat – Send Message
  • Owner: M5 (AI Lead) with M2
  • Dependencies: BE-PATPT-004 (ChatSession), BE-AI-007 (AI response), BE-AI-009 (Crisis detection), SignalR
  • Complexity: High
  • Estimated Effort: 6 hours

Objective

Implement the message sending functionality: store the patient's message, call the AI service to generate a response, store the AI response, and optionally detect crisis. Return both messages (or use SignalR for real‑time).

Business Purpose

Enable real‑time conversation between patient and AI assistant, with crisis detection.

Technical Purpose

Integrate with ChatService, AI service, and SignalR to provide a responsive chat experience.

Prerequisites

  • SignalR: Hub for real‑time messaging.
  • ChatService: Repository for ChatMessage.
  • AI Service: IConversationService (from BE-AI-007).
  • Crisis detection: ICrisisDetectionService.

Inputs

  • Session ID, message content.
  • Patient ID from token.

Outputs

  • AI response message (or sent via SignalR).
  • ChatMessage records saved.

Detailed Backend Workflow

  1. Create endpoint POST /api/patient/chat/send (or SignalR hub method).
  2. Validate that the session exists and belongs to the patient.
  3. Save patient message as ChatMessage with Sender = Patient.
  4. Call AI service to get response (passing conversation history, RAG context).
  5. Save AI message as ChatMessage with Sender = AI.
  6. Run crisis detection on the AI message (or patient message) – if crisis, create CrisisAlert and notify.
  7. Return or broadcast the AI message via SignalR.

Detailed AI Workflow

Patient Message → Save → RAG + History → GPT‑4o → AI Response → Save → Crisis Check → Notify

API Design

RouteMethodRequestResponseAuth
/api/patient/chat/sendPOST{ sessionId, message }{ aiResponse }Patient

Data Flow

Patient → POST /send → ChatService → AI Service → Crisis Detection → Response

Deliverables

  • Send message endpoint/hub method.
  • Integration with AI and crisis detection.
  • Unit tests.

Validation Checklist

  • ☐ Message saved correctly.
  • ☐ AI response generated and saved.
  • ☐ Crisis detection triggers alert if needed.
  • ☐ SignalR broadcast works.

Common Pitfalls

Pitfall: Not handling AI API timeouts – implement retry and fallback.
Pitfall: Forgetting to check session ownership – patient could send message to another patient's session.

Definition of Done

  • Send message works end‑to‑end.

Technical Purpose
