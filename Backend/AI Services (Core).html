BE-AI-001OCR Pipeline – Vision P0 High 8h ▾

Task Information

  • Task ID: BE-AI-001
  • Task Name: OCR Pipeline – Vision
  • Owner: M5 (AI Lead)
  • Dependencies: DB-014 (Vector Store Setup), BE-PAT-006 (Image Upload)
  • Complexity: High
  • Estimated Effort: 8 hours

Objective

Implement an OCR pipeline using GPT‑4o with Vision (or Azure Form Recognizer) to extract structured data from uploaded intake form images and store it in the IntakeForm entity.

Business Purpose

Automate data entry from paper intake forms, reducing manual effort and errors, and accelerating patient onboarding.

Technical Purpose

Integrate with OpenAI Vision API, handle image preprocessing, parse the response, and update the IntakeForm record with extracted data.

Prerequisites

Backend

  • OpenAI SDK: Understanding of OpenAIClient and vision capabilities.
  • Blob storage: Access to stored images.

AI

  • Vision API: GPT‑4o with Vision or Azure Form Recognizer.
  • Prompt engineering: Ability to craft prompts for structured extraction.

Inputs

  • IntakeForm ID (or image URL).
  • Image URL from blob storage.

Outputs

  • IntakeForm.ExtractedData populated with JSON.
  • Extracted fields (e.g., name, DOB, reason, medical history).

Detailed Backend Workflow

  1. Create OCR service: IOcrService with method ExtractFromImageAsync(string imageUrl).
  2. Implementation: Use OpenAIClient to call vision API with the image URL.
  3. Prompt: "Extract the following fields from this intake form: FullName, DateOfBirth, Gender, EmergencyContact, ReasonForVisit, MedicalHistory, Medications. Return as a JSON object."
  4. Parse response: Validate JSON structure.
  5. Update IntakeForm: Save extracted JSON.
  6. Error handling: If OCR fails, return error and allow manual fallback.

Detailed AI Workflow

Image URL → Vision API → JSON Response → Parse → Store in DB

API Design

Endpoint: POST /api/intake/{intakeFormId}/ocr – triggers OCR and returns extracted data.

Deliverables

  • IOcrService interface and implementation.
  • Controller endpoint for triggering OCR.
  • Unit tests with mocked OpenAI responses.

Validation Checklist

  • OCR returns valid JSON.
  • Extracted data is saved to IntakeForm.
  • Fallback works on failure.

Common Pitfalls

Pitfall: Image quality – low resolution images may cause extraction errors. Consider preprocessing or using a higher‑quality model.
Pitfall: Prompt not specific enough – test with diverse intake forms to refine prompts.

Testing Strategy

Unit Tests: Mock OpenAI API, test success and failure.

Integration Tests: Use a sample image and verify extraction.

Definition of Done

  • OCR pipeline works and stores extracted data.
  • Unit and integration tests pass.

Learning Resources

Must Know: OpenAI Vision API, prompt engineering.

Nice To Know: Azure Form Recognizer, image preprocessing.

BE-AI-004Embeddings & RAG – Setup P0 High 8h ▾

Task Information

  • Task ID: BE-AI-004
  • Task Name: Embeddings & RAG – Setup
  • Owner: M5 (AI Lead)
  • Dependencies: DB-014 (Vector Store), SessionNotes data
  • Complexity: High
  • Estimated Effort: 8 hours

Objective

Set up the embedding generation pipeline and RAG retrieval system: generate embeddings for session notes, store them in the vector database, and implement similarity search.

Business Purpose

Enable semantic search over session notes to provide contextually relevant information for report generation, chat, and summarization.

Technical Purpose

Integrate OpenAI Embeddings API, manage chunking, store vectors in pgvector, and query for similar chunks.

Prerequisites

  • OpenAI Embeddings: text-embedding-3-small or similar.
  • Vector store: pgvector with table for embeddings.

Inputs

  • SessionNote text content.
  • Patient ID, date, etc. for metadata.

Outputs

  • Embeddings stored in vector DB.
  • Ability to retrieve top‑k relevant chunks.

Detailed Backend Workflow

  1. Chunking: Split session notes into chunks (e.g., 500 tokens).
  2. Embedding generation: For each chunk, call OpenAIClient.GetEmbeddingsAsync.
  3. Store: Insert into Embeddings table with vector column and metadata (patientId, sessionId, chunkIndex).
  4. Retrieval: For a query, generate its embedding, perform cosine similarity search, return top‑k chunks.

Detailed Database Workflow

  • Table Embeddings: Id (uuid), ChunkText (text), Embedding (vector(1536)), Metadata (jsonb).
  • Index: ivfflat on embedding column.

Deliverables

  • IEmbeddingService – Generate embeddings.
  • IVectorStore – Add and query vectors.
  • Background job to index existing session notes.

Validation Checklist

  • Embeddings generated and stored.
  • Similarity search returns relevant chunks.

Common Pitfalls

Pitfall: Chunk size too large – affects precision. Use 500 tokens with overlap.
Pitfall: Not indexing the vector column – slow queries. Always create an index.

Testing Strategy

Unit Tests: Mock embedding API and test store/retrieval.

Integration Tests: Use a test vector store.

Definition of Done

  • RAG pipeline ready for use.

BE-AI-009Crisis Detection – Keywords P1 Medium 6h ▾

Task Information

  • Task ID: BE-AI-009
  • Task Name: Crisis Detection – Keywords
  • Owner: M5 (AI Lead)
  • Dependencies: BE-AI-007 (Chat responses)
  • Complexity: Medium
  • Estimated Effort: 6 hours

Objective

Implement crisis detection by combining keyword matching and GPT‑based sentiment analysis on chat messages to identify potential self‑harm or crisis language.

Business Purpose

Ensure patient safety by alerting therapists when a patient expresses distress or suicidal ideation.

Technical Purpose

Analyze each chat message, flag high‑risk content, and create a CrisisAlert record.

Prerequisites

  • Keyword list (crisis‑related).
  • OpenAI API for sentiment.

Inputs

  • Chat message content.

Outputs

  • Boolean – is crisis.
  • CrisisAlert entity created if detected.

Detailed Backend Workflow

  1. Check message for keywords (suicide, harm, hopeless, emergency, etc.).
  2. If keywords found, call GPT‑4o with a prompt: "Assess the risk level of this message: [message]. Return risk score (low/medium/high) and explanation."
  3. If risk is medium/high, create CrisisAlert.
  4. Notify therapist via notification service.

Data Flow

Message → Keyword Match → GPT Sentiment → CrisisAlert

Deliverables

  • ICrisisDetectionService.
  • Integration with chat pipeline.

Validation Checklist

  • Keywords trigger check.
  • Sentiment analysis returns risk.
  • Alert created when risk high.

Common Pitfalls

Pitfall: Over‑sensitivity – fine‑tune thresholds to avoid false positives.

Definition of Done

  • Crisis detection works.
